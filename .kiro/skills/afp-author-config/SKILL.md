# 产配置（Author Config）

AI 在 AFP 里的核心职责：产出可审计的确定性配置，交人审、交引擎确定性执行。**绝不直接产可运行代码或在运行期介入。**

> 路径说明：下文 `blocks/` `configs/` 指**构件仓这一角色**，按当前载体解析——验证期在 `experiments/expNN-*/src/` 下，生产期在仓库根。详见 `afp-core.md`「构件路径约定」。

## 何时用

- 用户要把一段需求落成一条装配流
- 新增/修改配置接线
- 已用 `afp-discover-blocks` 找齐可用装配块之后

## 步骤

1. **确认构件齐备**：列出本流用到的装配块（含版本）、转接件、数据来源。缺纯机制先回到 `afp-discover-blocks`。
2. **画接线**：声明"块 → 转接件 → 块"的连接结构、数据流向、枚举路由分支。
3. **静态可枚举自检**（红线，逐条过）：
   - 分支谓词是否是"已声明的有限域（布尔/枚举）"？是 → 留配置；"计算出来的/依赖运行时数据" → 改为封进装配块。
   - 循环是否"带类型集合的有界 for_each"？是 → 留配置；动态边界 → 入块。
   - 是否出现算法/深层嵌套条件？是 → 必须移出配置，进装配块当黑盒。
4. **契约对齐**：相邻块的输出 Schema 必须能喂进下游块的输入 Schema；不齐的地方用转接件做映射，而不是改块。
5. **产出 + 标注**：写出配置，并标注每个分支/循环为何"静态可枚举"，方便引擎做影响面分析、方便人审。
6. **交付前置**：提示该配置须过引擎校验 + 编译 + 归档测试才算交付。

## 红线

- 算法不入配置；配置只表达结构/接线/声明式策略。
- 配置必须保持"图可静态分析"（where-used / 死配置 / 影响面）。
- 引用的块/版本必须真实存在于 `blocks/`。
- 配置是设计期产物，运行期零 AI。
