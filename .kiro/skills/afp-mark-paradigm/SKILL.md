# 标记非 AFP 范式（Mark Paradigm）

项目第二阶段目标已升级（D-013）：不只验证 AFP 是否成立，而是**寻找 AI 介入模式下的更优编程范式**。验证期允许混合非 AFP 范式跑通——但每一处都必须**可机器扫描地挂牌**，否则范式边界会悄悄糊掉，最终的"适用域清单"（交付物 B）就无从谈起。

本技能是引入非 AFP 范式的标准动作。约定本体见 `afp-core.md`「范式混合标记约定」与路线图 `docs/paradigm-validation-sokoban-roadmap.md`。

**摆正心态（重要）**：标记非 AFP **不是认错、不是欠债**。项目的真实目标是找更优范式——非 AFP 在某处胜出是合法、甚至期待中的结论。`@paradigm` 的作用是让范式边界**可机器扫描、可诚实对比**，不是给非 AFP 判劣。判断它更优时，照实记，并按"红线"末条反向更新 `afp-core.md`。

**只标该标的（别过度贴标签）**：标记针对"读者会默认为纯 AFP 数据流的业务/装配逻辑"里的偏离处。**引擎实现、UI/渲染、测试脚手架、构建脚本不需标记**——它们本就不在 AFP 承诺范围内。判据：不标记会不会让人误以为"这段是纯 AFP"？会才标。

## 何时用

- 某段逻辑用 AFP 的无状态数据流（DAG）表达不了：需要跨调用记忆、动态边界循环、时间维度状态（撤销栈）、可变共享状态等。
- 要引入 reducer / 状态机 / 有状态块 / 全局状态 / 类继承 / ECS / 行为树等非 AFP 结构。
- 已落地实例可参照：`experiments/exp04-k-state/`（MVP-0 方案 B 的有状态块）。

## 步骤

1. **先问能否不引入**（关键，别急着挂牌）：
   - 这段状态能不能改成 AFP 方案 A——由调用方在回合间持久化、作 `initialInput` 进 / output 出，块保持纯？
   - 这个"分支/循环"是不是其实是已声明有限域的枚举路由 / 有界 for_each（那就留配置，不算非 AFP）？
   - 能留在 AFP 就别引入非 AFP——这是"最小范式"原则（不滥用），**不是**因为非 AFP 低人一等。确实更适合非 AFP 的，大方用、如实标。
2. **选最小范式**：确实要引入时，选能解决问题的**最小**非 AFP 结构，不顺手加大。给它起一个简短范式名（`reducer` / `state-machine` / `stateful-block` / `global-state` ...）。
3. **打三段标记**：在该代码文件头部加统一标记块：
   ```ts
   /**
    * @paradigm NON-AFP: <范式名>
    * @reason <为什么 AFP 在这里不合适——具体到这段逻辑，不要套话>
    * @afp-debt <未来可能的收编方向：归入转接件 / 升级为引擎一等概念 / 待 MVP-N 复审>
    */
   ```
4. **隔离与对称**：把非 AFP 部分收拢在尽量小的边界内（一个块工厂 / 一个 runner），别让记忆/可变状态渗进周边纯块。若同时存在 AFP 对照方案，保持两者**失败语义对称**（如非法输入两路都在同一处拦下、状态都不前进），让对比只反映范式差异。
5. **配测试 + 验证**：非 AFP 代码同样要测；按 `test-and-acceptance.md` 验证期最低门禁跑 `typecheck` + `test`（+ 端到端入口）得真实输出。有状态块的"reset/隔离"成本如实测出来。
6. **登记证据，供交付物 B**：把"这里用了什么范式、为什么、代价是什么"写进当期实验/MVP 的 REPORT；必要时在 `docs/open-questions.md` 对应议题下补"实现期发现"。这些是最终 `docs/paradigm-comparison.md`（范式适用域清单）的原料。

## 红线

- **没打标记的非 AFP 代码 = 违规。** `grep "@paradigm"` 必须能扫到全仓每一处非 AFP。
- 标记不是为了"合法化随意引入"——第 1 步"能否不引入"先过，留得在 AFP 就不挂牌。
- 验证期**不强行**把它归类为转接件；先如实记录它实际是什么，MVP 跑完再回头评估能否收编。
- 不为收尾粉饰：非 AFP 引入暴露的别扭、缺口、成本，照实记，失败结论与成立结论同等有价值。
- 某非 AFP 方案被证明明显优于纯 AFP 时，反向更新 `afp-core.md`，明确"AFP 在 X 场景应让位于 Y"——这才是本阶段真正的产出。
