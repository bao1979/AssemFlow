# 装配块纯度与内容外置规范

> 这是 `afp-core.md`"机制/策略二分"原则的可执行编码细则，写装配块/转接件代码时自动加载。
> 核心命题：装配块一旦硬编码了业务内容，就不再是纯机制，全球复用立刻死掉。

## 核心原则

**装配块只负责纯机制，不含任何业务内容。** 业务内容通过配置（Config）与数据（Data）从外部喂入，装配块通过参数/输入 Schema 获取。

## 装配块（Block）里禁止硬编码

- 面向用户的文案（提示语、错误提示、邮件正文模板等）
- 业务数据值（具体的阈值、价格、配额、白名单条目）
- 业务规则与策略判断（"VIP 享 8 折"这类策略）
- 外部资源地址（URL、桶名、队列名、密钥）
- 任何与某个具体业务绑定的标识

> 出现以上任意一条，说明这段东西不是纯机制——要么提为输入参数（走配置/数据），要么它根本是业务策略（走转接件/配置）。

## 装配块里允许存在

- 纯机制算法与控制流（这正是"算法入块"的归宿）
- 错误码字符串（如 `"email_invalid"`）——这是契约，不是用户文案
- 开发者日志
- 代码注释（中文）

## 内容/数据的归属

| 类型 | 归属 | 说明 |
| :--- | :--- | :--- |
| 业务文案 / 模板 | 配置或数据 | 装配块只收"已渲染好的值"或"模板 + 数据"作为输入 |
| 阈值 / 配额 / 价格 | 配置 | 作为声明式参数注入 |
| 业务规则 / 策略路由 | 配置（枚举路由）或转接件 | 见 `afp-core.md` 静态可枚举红线 |
| 外部资源地址 / 密钥 | 环境变量 / 数据 | 不进任何块或转接件源码 |
| 运行时 / 测试数据 | `data/` | 通过输入传入，不在代码里 `readFile` 写死路径 |

## 转接件（Adapter）的纪律

转接件允许承载"不可再约简的业务决定"，但同样：

- 反复出现的字段映射/类型转换不要留在转接件——用 skill `afp-extract-block` 挖成装配块。
- 转接件里也不要硬编码外部资源地址与密钥（走环境变量/数据）。

## 自检清单（写完一个块/转接件回看）

1. 把这个块原样搬到另一个完全不同的业务，能否不改一行代码就用？不能 → 有业务内容泄漏进来了。
2. 块里有没有出现具体业务名词、文案、URL、魔法数字？有 → 外置。
3. 块的输出是不是只依赖输入（确定性）？依赖了全局状态/时钟/随机？→ 改为显式输入，否则属性测试会抓到。
