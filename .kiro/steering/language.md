# 语言规范

## 对话与工具调用

- 与用户的所有对话、说明、摘要、状态汇报必须使用中文
- 工具调用之间的过渡性说明、思路描述、计划陈述也必须使用中文（这些是与用户的沟通，不是代码）

## 代码

- 变量名、函数名、类名、文件名保持英文
- 注释使用中文

## 文档（给 Plucker518 看的）

- 正文、标题、表格内容、描述性文字必须使用中文
- 只有以下情况允许使用英文：
  - 会出现在代码或程序中的专有名词（如 CSS Variables、CSS Modules、Next.js、SVG、API、Token、Props）
  - 文件路径和代码片段（如 `src/styles/tokens.css`、`--cw-bg-primary`）
  - 已成为行业通用缩写且无合适中文对应的术语（如 OG Image、SSR）
- 能用中文表达的概念不用英文。例如：用"基础层"不用"Foundation"，用"体验增强"不用"Experience"，用"可折叠"不用"Collapsible"
- 组件名在文档中首次出现时用中文描述 + 括号标注代码名，后续可直接用代码名。例如："全局导航栏（NavBar）"

## 对外展示文档（双语例外）

面向 GitHub / 全球受众的对外文档（仅限仓库根 `README.md` 及其英文版）是上述"文档必须中文"规则的**例外**：

- 允许英文/双语，以便全球展现。
- **中文是唯一内容真相源**；英文版（`README.en.md` 或对调后的 `README.zh-CN.md`）是**翻译派生视图**。
- 英文版头部须标注"译自中文版"。**改了中文门面，必须同步重新生成英文版**，否则英文视为过期。
- 此例外**不适用**于内部文档（`docs/ai/*`、`docs/` 下分析文档、steering、skills）——它们一律中文。

## 文档章节标题

- 禁止使用中文数字编号（一、二、三…）作为章节序号——容易在插入/删除时错位
- 使用纯描述性标题，不带编号。例如：`## 体验流程`、`## 成败标准`
- 文档内部靠 Markdown 标题层级（##、###）表达结构，靠标题文字本身表达内容
- 如果确实需要有序列表（步骤、优先级），用正文中的阿拉伯数字列表（`1. 2. 3.`），不放在 `##` 标题里
