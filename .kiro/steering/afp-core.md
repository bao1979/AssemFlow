# 装配流编程（AFP）核心纪律

> 本文件是 AssemFlow 项目的"宪法"，所有设计取舍以此为准。
> 理论依据：`docs/装配流编程-可行性分析.md`。本文件只沉淀"必须遵守的纪律"，论证不在此重复。

## 一句话定位

AssemFlow 不是"AI 编程助手"，而是"业务蓝图的 CAD 工具 + 确定性编译器"。
**AI 负责发现与配置，引擎负责确定性执行。**

## 五元构件（术语以此为准）

| 构件 | 英文 | 身份 | 变化速率 | 谁维护 |
| :--- | :--- | :--- | :--- | :--- |
| 装配块 | Block | 纯机制，与业务无关的最小组件 | 几乎不变 | 全球共享，像 lodash |
| 转接件 | Adapter | 业务适配 / 防腐层，按需分流 | 随业务变 | 各业务自己写 |
| 配置 | Config | 接线蓝图，声明式策略 | 经常变 | 设计期 AI 产、人审 |
| 数据 | Data | 运行时 / 测试数据 | 时刻在变 | 业务方提供 |
| 装配流 | Flow | 由上四者组合而成的完整业务流 | — | 配置定义 |

> "原子流"是旧词，一律指装配块（Block）。

### 构件路径约定（载体随阶段走，纪律不变）

五元构件的**身份与纪律**与它存放在哪无关；路径只是载体，随阶段浮动：

| 构件 | 验证期（实验内） | 生产期（仓库根） |
| :--- | :--- | :--- |
| 装配块 | `experiments/expNN-*/src/blocks/` | `blocks/` |
| 转接件 | `experiments/expNN-*/src/adapters/`（按需） | `adapters/` |
| 配置 | `experiments/expNN-*/src/configs/` | `configs/` |
| 数据 | 实验内就近 / `data/` | `data/` |

skill 与文档里写 `blocks/` `adapters/` `configs/` 时，指的是**构件仓这一角色**，不是某个固定目录——检索/落地时按当前载体解析到上表对应位置。这条是 SSOT，其它文件引用它、不重述。

## 不可逾越的硬边界（违反即范式退化）

1. **AI 只在设计期产配置，运行期拼装零 AI。** 这是确定性能成立的唯一前提。一旦为了图快让 AI 直接产代码或运行期介入，AFP 立刻退化成 vibe coding。
2. **算法不入配置。** 配置只准表达"结构、接线、声明式策略"；算法必须留在装配块（代码）里。在 YAML/JSON 里写算法 = 造一门很烂的编程语言。
3. **配置图必须静态可枚举。**
   - 允许留在配置：对已声明的有限域谓词（布尔/枚举）做分支、对带类型的集合做有界 `for_each`。
   - 必须封进装配块当黑盒：计算出来的 / 依赖运行时数据的谓词、深层嵌套条件、动态边界循环。
   - 判据："VIP 走 A 否则走 B"这类对枚举的简单路由留配置；复杂判定入块由单测覆盖。
4. **依赖白名单 + 制品校验。** 装配块的依赖必须在白名单内、可审计，从源头掐死 slopsquatting（AI 幻觉包名投毒）。
5. **生成产物必须过编译 + 跑归档测试才算交付。** 杜绝"看起来对"的幻觉代码。
6. **落入"10% 区"不硬装。** 高度定制算法、复杂 UI、强耦合遗留改造、需求长尾——回退到人工/对话式开发，并引导沉淀为新装配块。

## 配置格式纪律

- **配置源必须是纯数据格式：JSON（开发期可用 JSONC 允许注释）+ JSON Schema 校验。** 选纯数据格式是为了从物理形态上堵死"算法入配置"——纯数据塞不进可执行逻辑。
- **禁止用可内嵌逻辑的格式做配置源**（如 HTML 可嵌 `<script>`、JS 对象会求值）。HTML 在 AFP 里只能作为引擎**派生的可视化视图**（把配置图渲染成图给人审/展示），不能作为配置的源。
- **配置的"显示"是引擎的职责，不是配置格式的职责。** 配置源服务于机器分析与人审，可视化由引擎渲染派生。

## 三条派生原则（设计时的取舍依据）

1. **机制/策略二分。** 装配块是纯机制（可全球复用），业务的善变锁进转接件（局部、廉价、可丢弃）。"用户注册"不是装配块，它是装配流。
2. **配置是可被机器导航的债。** 配置天生是一张图：where-used、死配置检测、影响面波及、类型校验都能从配置结构直接算出。**从第一天就把依赖图谱 / 影响面分析当作引擎一等公民**——代码债看不见，配置债看得见，这是要兑现的优势。
3. **治理关节，放开零件。** 只钉死契约（I/O Schema、版本、转接协议），让粒度自由浮动。目标是"切错了能廉价改对"，而非"一次切对"。拿不准时切细（组合可逆、分解破坏性）。

## 收敛纪律（防止熵从河床搬到闸口）

- 转接件**只用组合/模板化，禁止继承**（继承=脆弱基类，会把刚甩掉的耦合请回来）。
- 持续把转接件里反复出现的"字段映射/类型转换"挖出来，**升级成装配块**（它们本就是纯机制）。转接件应越缩越小，最后只剩不可再约简的业务决定。
- 对转接件套用与装配块同一套重复检测回路。

## 版本与演化

- v1.0 前鼓励激进重构；v1.0 后走语义化版本，引擎必须支持**别名 / 多版本共存**，新旧粒度并行。
- 装配块是纯机制 + 强契约，schema diff 机械可算，引擎**或许**能 codemod 把配置从 `block@v1` 迁到 `v2`。

> ⚠ **本节是待验证假设，不是已证结论。** 可靠演化是 AFP 目前最薄弱、最未经证明的一环：自动迁移在语义级 breaking change 上未必成立、多版本共存可能版本碎片化、特性交互（组合后变坏）尚无解。**在实验证明之前，不得把"可靠演化"当作既定能力对外宣称或据此做设计承诺。**

## 生态愿景修正（别撞 30 年那堵墙）

- 能全球复用的只有**装配块（纯机制）**；转接件/配置/数据是局部资产，天然不全球复用。
- 因此共享的不是"开箱即用的业务流"，而是 **「装配块 + 转接模式 + 配置模板 + 起手式」**（可复用的起点，不是成品）。
- **先有杀手级引擎，再谈注册表。** 单机对单一用户有用先于多人有用。不要让"生态冷启动"绑架"验证冷启动"。

## 范式混合标记约定（验证期）

> 背景：项目目标已升级（D-013）——从"证明 AFP 成立"转为"寻找 AI 介入模式下的更优编程范式"。
> 验证期为跑通**允许混合非 AFP 范式**（reducer / 状态机 / 有状态块 / 全局状态 / ECS / 行为树等），但**必须可机器扫描地标识**。
> 全局路线图与跨 MVP 语境见 `docs/paradigm-validation-sokoban-roadmap.md`；本节是它在"写代码时自动在场"的纪律沉淀。

凡**会被读者默认为 AFP 的业务/装配逻辑**里掺入了非 AFP 范式，该代码文件头部加统一标记块：

```ts
/**
 * @paradigm NON-AFP: <范式名>   // reducer / state-machine / stateful-block / global-state ...
 * @reason <为什么 AFP 在这里不合适>
 * @afp-debt <该选择的权衡与"未来重审/可能收编"备注；若判断非 AFP 才是对的，照实写，不必假装要偿还>
 */
```

**标记的目的是"现形"与"诚实对比"，不是给非 AFP 判刑。** 这条服务于 D-013 的真实目标——找更优范式。非 AFP 在某处胜出是合法结论，不是失败；`@afp-debt` 是权衡记录，不是"欠 AFP 的债必须还"。

适用范围（避免过度贴标签）：
- **要标记**：装配流/业务逻辑层里偏离 AFP 数据流的部分（有状态块、reducer、loop step、全局状态等）。
- **不需标记**：引擎实现本身、UI/渲染层、测试脚手架、构建脚本——它们本就不在"配置即图"的 AFP 承诺范围内，贴标签只会制造噪声。判据：**一个读者会不会因为没标记而误以为"这段是纯 AFP 数据流"？会 → 标记；不会 → 不标记。**

约束与流程：
- `grep "@paradigm"` 必须能一键列出业务/装配逻辑里所有非 AFP 处。
- 引入前先问"能否留在 AFP"——能留就别引入（最小范式原则）；这是为了不滥用，不是因为非 AFP 低人一等。
- 验证期**不强行**把它归类为转接件；先如实记录它实际是什么，跑完再回头评估能否收编。
- 引入非 AFP 范式的标准动作见 skill `afp-mark-paradigm`。
- 某非 AFP 方案被证明明显优于纯 AFP 时，该结论**反向更新本文件**，明确"AFP 在 X 场景应让位于 Y"——这正是本阶段要的产出。

## 当前阶段约束

- 项目处于**研究稿 v0.1 验证阶段**，非实施承诺。
- 三个最小证伪实验（甜区/边界/复用）是否启动，属待 Plucker518 决定的开放问题；在获明确许可前不擅自铺开实验代码或引擎实现。
- clawerworld 仅作"边界试验田 + 真实素材来源"，**不是生态供给起点**，也不做甜区证明试验田。
