# Requirements Document

> Spec: sokoban-mvp-0-k-state · MVP-0 of the Sokoban paradigm-validation chain.
> 全局共识、术语、Godot 参照、范式标记约定见 `docs/paradigm-validation-sokoban-roadmap.md`。

## Introduction

这是 Sokoban 范式验证链的第 0 站，对应 `open-questions.md` 的 **Q-026（K-STATE）**。

在碰游戏之前，先用一个**与游戏无关的最小状态机**（红绿灯）定下一件事：长寿命对象的状态怎么存、怎么演化。把这个共识立为**后续 MVP 的默认起点**（阶段性默认，**非终局定案**——MVP-1 的大体量状态若暴露反例，允许复审改换），避免一上来同时碰多个未知。

**变量隔离原则**：状态承载方案 A/B 的真实差异分布在两个轴上——(1) 状态体量（全量状态每回合进/出是否笨重），(2) 迁移复杂度（迁移是否依赖外部输入）。MVP-0 **只取轴 2**：状态机带最小外部输入，使迁移成为 `下一状态 = f(当前状态, 输入)`（与 Sokoban `下一状态 = f(当前网格, 方向键)` 同形），但状态体量刻意保持极小（单枚举）。轴 1（大体量状态）留给 MVP-1 的网格。这样 MVP-0 既最小又不退化——纯循环状态机会让 A/B 几乎无差，结论会结构性偏向 A，误导后续 MVP。

非 AFP 范式（若引入）须按路线图的"范式混合标记约定"打 `@paradigm` 标记。

## Glossary

完整术语见 `docs/paradigm-validation-sokoban-roadmap.md` 的 Glossary。本 MVP 关键词：

- **K-STATE**：长寿命对象的状态演化议题，对应 Q-026。
- **状态承载方案 A / B**：A = 调用方持久化（状态作 initialInput 进 / output 出，块保持纯）；B = 运行时承载状态（状态存活在「引擎所运行的有状态块」内，配置只接线输入；对应 Q-026 设想的"给 BlockDef 加 state / 有状态块"）。
- **外部输入**：每回合喂给状态机、能改变迁移路径的信号（红绿灯里如"行人请求"；Sokoban 里即方向键）。

## 背景：状态放哪——方案 A vs 方案 B

本 MVP 的核心议题只有一句话：**长寿命对象的"当前状态"存在哪里。**

**方案 A · 调用方持久化（状态在外，逻辑纯）**

```
状态(引擎外) ──作 initialInput 进──> [纯装配流] ──nextState 出──> 调用方存起来
                                       ↑ 引擎/块无状态、纯函数
```

引擎与块都无记忆，`下一状态 = f(当前状态, 输入)`。状态由外部（调用方 / 前端 / 数据库）在两次调用之间保管，每回合再喂进来。引擎永不"记住"任何东西。

**方案 B · 引擎承载状态（状态在内）**

```
调用方只发"输入" ──> [引擎持有状态存储] ──> 引擎自己读旧状态、算新状态、写回
                          ↑ 引擎内有一块可变状态区
```

引擎内部有状态存储，块通过它读写。调用方只发输入，不关心状态在哪，引擎负责跨回合的记忆与恢复。

### 为什么必须比这一对

AFP 的根基是**无状态、确定性的数据流（DAG）**：配置=接线、块=纯机制、同输入同输出，天生擅长"一次性数据变换"。但游戏角色、订单、红绿灯这类**长寿命对象**有活过单次调用的状态，逼出一个 AFP 未回答的问题：这块状态放哪才不破坏 AFP 的纯粹性？

- 选 A：块继续纯，代价是调用方每回合穿进穿出**全量状态**——状态体量大时（如 Sokoban 网格）配置可能笨重啰嗦。
- 选 B：调用方轻松，但引擎里多了一块**可变状态**——这正是 AFP 想躲开的，可能让"配置即图、一眼可审"的卖点变模糊。

MVP-0 比的不是"谁赢"，而是**A 的全量状态穿透有多痛、AI 还容不容易照着产配置**。这也是本 spec 给红绿灯加外部输入、又刻意保持状态极小的原因（见上文变量隔离原则）。

### 现实参考（已跑成的工业先例）

"状态放哪"在工业界有大量成熟先例，且方案 A 一侧的成功案例尤其硬：

| 阵营 | 代表项目 / 模式 | 对应关系 |
| :--- | :--- | :--- |
| **A**（纯转移 + 外部持有） | Elm Architecture（`update(Msg, Model)->Model`） | 框架持 Model、update 纯，A 的教科书形态 |
| **A** | Redux（`reducer(state, action)`） | reducer 纯，store 只是哑容器 |
| **A** | 确定性 lockstep 游戏模拟（星际/帝国时代网络同步） | `nextState = step(state, inputs)`，与回合制 Sokoban 几乎同构——**最贴的参考** |
| **A** | Event Sourcing | 当前状态 = 事件日志 fold，状态派生、外部持久化 |
| **A** | AWS Lambda / FaaS | 函数无状态，状态在外部存储，每次载入 |
| **B**（运行时持有） | Temporal / AWS Step Functions | 引擎跨步骤持久化、断点续跑——**B 最成功的样板，但引擎极重** |
| **B** | Actor 模型（Erlang / Akka） | actor 私有可变状态，运行时管状态 |
| **B** | Godot / Unity Node 场景树 | Node 自带状态、引擎 tick 改它——**正是路线图明确拒绝的架构** |

### 预判（供 Design 阶段参考，非结论）

Redux/Elm、确定性 lockstep、event sourcing 这些最经得起时间的范式几乎一边倒站方案 A。方案 B 也能成（Temporal 是铁证），但代价是一个会做持久化与恢复的重引擎，而那种引擎里的可变状态恰是 AFP 想躲的。故预判 MVP-0 大概率落在 A——但实验真正产出是"A 的全量状态穿透多笨、AI 友不友好"，以及诚实记录"在什么临界点 B 才开始划算"。预判不替代 Requirement 2 的实测对比。

## Requirements

### Requirement 1: 带外部输入的红绿灯状态机最小实现

**User Story:** 作为研究者，我希望先用一个与游戏无关、但带最小外部输入的状态机，定下"长寿命对象的状态怎么存、怎么演化"，再进入游戏——避免一上来同时碰多个未知，同时不让实验退化到"纯循环"而失去信号。

#### Acceptance Criteria

1. WHEN MVP-0 启动 THEN 系统 SHALL 实现一个红绿灯状态机：状态在 tick 与行人请求两种外部输入下沿红→绿→黄→红推进，且至少存在一个外部输入能改变迁移路径——具体为绿灯设为驻留态（tick 保持绿），仅当收到行人请求才切黄，使绿灯下 tick 与 pedestrian 产出不同的下一状态。独立于 Sokoban。
2. WHEN 给定当前状态与一个外部输入 THEN 系统 SHALL 确定性地算出下一状态（同状态 + 同输入永远同结果）。
3. WHEN 实现状态机 THEN 状态体量 SHALL 刻意保持极小（单枚举），不引入大体量状态——后者是 MVP-1 网格要验证的变量，本 MVP 不碰。

### Requirement 2: 状态承载方案对比

**User Story:** 作为研究者，我希望对比"调用方持久化"与"引擎承载状态"两种方案，选出后续统一采用的状态承载方式。

#### Acceptance Criteria

1. WHEN MVP-0 完成 THEN 文档 SHALL 记录两种状态承载方案的对比："A. 调用方持久化（状态作 initialInput 进 / output 出）"与"B. 引擎承载状态"，对比维度为：配置可读性、**配置对 AI agent 的友好度预估（推测性分析，非实测）**、调试容易程度。
2. WHEN 记录"配置对 AI agent 的友好度"维度 THEN 文档 SHALL 明确标注此维度为推测，不下实测结论；真正的 AI 实测整体留给路线图交付物 A（`docs/agent-test-prompts.md`）。
3. WHEN MVP-0 给出结论 THEN 该结论 SHALL 作为后续 MVP 的**默认起点**（阶段性默认，**非终局定案**）；MVP-1 大体量状态若暴露反例，SHALL 允许复审改换——不得把窄切面结论伪装成终局。
4. WHEN MVP-0 给出结论 THEN 文档 SHALL 诚实标注适用边界：本状态机虽含外部输入但状态体量极小，"大体量状态下 A/B 的差异"未在本 MVP 验证，留待 MVP-1 网格。
5. IF 两种方案都不令人满意 THEN 项目 SHALL 如实记录并探索第三方案（如 reducer），不允许跳过 MVP-0 直接做游戏。
