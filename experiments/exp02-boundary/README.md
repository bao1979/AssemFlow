# 实验② · 边界证伪

> 目的不是证明 AFP 能行，而是找到它**在哪崩**、崩得多难看。

## 要证明什么

一句话：**AFP 处理"条件逻辑密集 + 步骤间强耦合"的定制需求时，会在哪一步被迫违反自己的纪律？**

实验①②③分工：
- 实验①证明甜区（线性流）能行 ✅
- 实验③证明封装+复用能行 ✅
- **实验②要找出"不行"的地方**——划清边界

## 选什么需求来测

需要满足三个条件：
1. **真实**（不是编造的难题）
2. **偏定制**（不是实验①那种纯线性）
3. **能逼出条件爆炸 / 步骤间强耦合 / 动态路由**

选：**订单折扣计算**。

场景：一个电商下单流程的"算折扣"环节——
- 多种折扣规则可叠加（满减、会员折、新人券、限时促销）
- 规则之间有优先级和互斥（满减和新人券不能同时用）
- 最终折扣取决于运行时数据（购物车金额、用户历史、当前时间）
- 规则会频繁变更（业务想随时加/删/改规则）

这个场景精准命中"10% 陷阱"：规则组合是条件爆炸、优先级和互斥是算法、依赖运行时数据的谓词。

## 预设崩点假设（事先写下，实验后对照）

| 假设 | 预期结果 |
| :--- | :--- |
| 折扣规则的优先级/互斥判断是算法 | 按纪律不能入配置，必须进块 → 但这让"业务想随时改规则"变难 |
| "哪些规则生效"依赖运行时数据 | 不满足"静态可枚举"红线，不能留在配置里路由 → 必须封进块当黑盒 |
| 规则频繁变更 | 如果规则在块里，每次改规则就要改代码 → 违反"只改配置就能改行为"的甜区假设 |
| 多规则叠加的组合效应 | 特性交互问题：单个规则正确，组合起来可能冲突 → 配置图静态分析管不了 |

核心矛盾：**业务想把规则放配置里（频繁改、不碰代码），但纪律要求算法不入配置。** 这是 AFP 的根本张力。

## 怎么跑

1. 先尝试"纯 AFP 方式"：把折扣计算拆成块+配置，看配置会膨胀/退化到什么程度
2. 记录每一步的感受："到这里还行""到这里开始别扭""到这里彻底崩了"
3. 如实记录崩点：是配置退化成 YAML 地狱？还是频繁改块代码？还是引擎能力不够？

## 成功/失败判据

这个实验**没有"成功"**——它的价值就是找到边界并诚实记录。

判据是：**是否清晰地划出了 AFP 的适用边界线，并给出"碰到这类需求该怎么办"的建议。**

## 不做什么

- 不为了让 AFP 能行而硬凑解决方案（那是粉饰）
- 不连真实电商系统（内存模拟足够暴露结构问题）
- 不做完整的电商下单（只做"算折扣"这一个环节）

## 实际结论

### 崩点定位（从"还行"到"崩了"的梯度）

| 阶段 | 什么能用 AFP | 什么不能 |
| :--- | :--- | :--- |
| ✅ 还行 | check-threshold（阈值判断）、apply-discount（打折）、check-exclusion（互斥检查）作为纯机制块完全没问题 | — |
| ⚠️ 别扭 | 互斥对定义可以作为"数据"传入块（而非硬编码）——这是转接件/数据层的活 | 互斥算法本身必须在块里，改互斥规则 = 改数据输入（而非改块代码），勉强可接受 |
| ❌ **崩了** | — | **规则的条件（condition）是函数**——`(order) => order.amount >= 200`。函数不能放 JSON 配置（物理上塞不进），必须在代码里。但业务想频繁改条件 → AFP"只改配置就能改行为"的甜区假设在这里失效 |

### 根本矛盾

> **配置不准放算法 vs 业务想在配置里改规则。**

这不是 AFP 的 bug，而是它刻意选择的取舍：守住"配置不退化成烂语言"的纪律，代价是"条件密集的策略调整不在甜区内"。

### 应对建议（诚实的、不粉饰的）

1. **如果条件是有限枚举（如 userType ∈ {normal, vip}）**：可以留在配置——对枚举的简单路由是允许的。
2. **如果条件依赖运行时计算（如 amount >= 某个动态阈值）**：封进块，业务改条件 = 改块代码 + 发版。AFP 在这里退化成"改代码"，不再是"只改配置"。
3. **混合方案（AFP 的实际边界线）**：把"条件判断"抽成一个"规则引擎块"——块的输入是"一组规则描述（DSL 或数据）"，块内部用 DSL 解释器执行。这样"改规则 = 改输入数据"而非改块代码。但这等于在块里藏了一个小型规则引擎——复杂度没消失，只是从"配置地狱"搬到了"块内 DSL"。

### AFP 适用边界（实验②划出的线）

| 在线内（AFP 甜区） | 在线外（AFP 力不从心） |
| :--- | :--- |
| 步骤间是线性/枚举路由 | 步骤本身的结构取决于运行时数据（动态管道） |
| 策略变更 = 切换枚举值或调参数 | 策略变更 = 修改条件逻辑/算法 |
| 纯机制块可全球复用 | 条件密集的业务规则频繁变 |
| 少量规则，组合可穷举 | 多规则叠加，组合爆炸，特性交互 |

### 局限

- 只测了"折扣计算"一种边界场景，其他类型（复杂 UI 交互、强耦合遗留改造）未碰
- 没用引擎跑（用的是手写拼装），引擎的限制可能暴露更多问题
