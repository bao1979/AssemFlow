# 实验③ · 封装 + 复用证伪

> 先定判据，再写代码。

## 要证明什么

两个假设，分两步验证，信号分开：

**假设 A（封装）**：能廉价地把 npm 现成的纯函数封装成合规装配块（配契约 + 薄壳 + 过属性测试）。

**假设 B（复用）**：同一个"来自外部库"的块，能跨两条不同的装配流复用——不改块代码，只加配置/转接件。

## 怎么算成功、怎么算失败

**假设 A 成功**：封装出的块满足——
1. 有 TypeBox 写的 I/O Schema（标准 JSON Schema 产物）
2. 薄壳代码 ≤ 20 行（不算注释）
3. 属性测试（确定性：同输入同输出）通过

**假设 A 失败**：如果某类函数封不成合规块（如必须带副作用、或签名太复杂无法用 Schema 表达），如实记录崩点。

**假设 B 成功**：同一个块被两条不同的流引用，两条流各自跑通，且块代码零修改。

**假设 B 失败**：复用时必须回头改块代码。

## 选什么库封装

你的要求：**用最广范围的**。

lodash 是 npm 下载量最大的工具库（周下载 5000 万+），函数种类覆盖面最广。用它代表"如果 lodash 能封，绝大多数纯函数库都能封"。

从 lodash 里挑 3 种代表性签名：

| 函数 | 签名特征 | 选它的原因 |
| :--- | :--- | :--- |
| `_.capitalize` | 单参数纯函数 | 最简单的情况，封装应该是零成本 |
| `_.get` | 多参数（对象 + 路径 + 默认值） | 常见的多参工具函数 |
| `_.now` | 零参数、读系统时钟 | **反例**——不是纯函数，预期封装应该失败或需要改造 |

第三个故意选非纯的，目的是让实验诚实地标出"哪类函数封不了"。

## 两条流怎么设计

- **流 A**：对用户输入做"首字母大写 + 从嵌套对象里取值"（用封装好的 capitalize 块 + get 块）
- **流 B**：对另一种数据做"首字母大写 + 从嵌套对象里取值"（同样的块，不同的配置参数）

两条流用同一批块、不同配置，验证复用。

## 不做什么

- 不在这个实验里造"自动封装工具"——手动封装，验证的是"封装这件事本身是否廉价"。
- 不碰 lodash 的高阶函数（如 `_.map` 带回调）——那涉及"行为参数"问题，是更复杂的课题，留后续。
- 不连真实服务，全部内存跑。

## 工程

- 独立小工程 `experiments/exp03-wrap-reuse/`
- 依赖：lodash + 实验①同款开发工具（TypeScript/vitest/fast-check/TypeBox/Ajv）
- 依赖白名单 allowlist.json 登记

## 实际结论

**假设 A（封装）为真。** lodash 纯函数能廉价封装成合规装配块：
- 薄壳 1-2 行，总计含 Schema 约 15 行/块
- 属性测试（确定性）通过
- 反例 `_.now`（非纯）被属性测试自动挡住——AFP 纪律的工程价值得到验证

**假设 B（复用）为真。** 同一批块跨两条不同的流复用，零修改块代码，各自跑通。

局限：
- 只测了简单签名（单参/多参），没碰高阶函数（带回调的如 `_.map`）
- 没碰有副作用的 I/O 类库（如 fs、http）
- `Type.Unknown()` 处理宽泛签名可行但牺牲了 Schema 校验深度
