# 实验① · 甜区证伪

> 这是实验方案。按纪律，开写代码前先把"要证明什么、怎么算成功/失败"定下来。

## 要证明什么

一句话假设：

> **在"注册"这类结构稳定的流程上，只改配置就能拼出能跑的软件，不用改任何零件（装配块）。**

这是 AFP 最核心的卖点。本实验用最小代价验证它到底成不成立。

## 怎么算成功、怎么算失败（先定死，不许事后改）

- **成功（假设为真）**：手写几个与业务无关的零件 + 一份配置，拼出注册流并跑通；然后**只改配置**（把通知方式从"邮件"换成"短信"），不碰任何零件代码，就能改变行为且仍然跑通。
- **失败（假设被证伪）**：如果"只改配置"做不到——必须回头改零件代码，或拼出来的东西跑不起来——就如实记录：在哪一步崩、为什么崩。**失败也是有价值的结论，绝不粉饰。**

## 不做什么（防止跑偏加料）

- 不连真实数据库、不发真实邮件——全部用内存里的"假"实现，保证每次结果一样。
- 不做密码强度校验、限流、花哨的错误处理——这些跟"证明假设"无关。
- 不在这个实验里造引擎——用一段手写脚本模拟"引擎照配置拼装"的动作。

## 用什么例子：注册流

把"用户注册"拆成几个**与业务无关的小零件**（装配块），再用一份配置把它们接起来。

## 拆成哪些零件（装配块）

| 零件 | 干什么 | 为什么它是"纯机制" |
| :--- | :--- | :--- |
| 查邮箱 | 判断邮箱格式对不对 | 跟哪个业务无关，谁都能用 |
| 加密密码 | 把密码哈希后再存 | 同输入永远同输出（盐当输入传进来） |
| 存用户 | 把用户存起来 | 存到内存，状态当参数传入，不用全局变量 |
| 发通知 | 发一条通知 | 只定义"怎么发"，发到哪由配置选 |
| 记审计 | 追加一条审计记录 | 纯粹追加，不掺业务判断 |

## 配置长什么样

一份 JSON，声明这条流怎么接：`查邮箱 → 加密密码 → 存用户 → 发通知 → 记审计`。
其中"发通知"用**枚举选渠道**（`email` 或 `sms`）——简单二选一路由，允许留在配置里。

## 怎么验证（冒烟）

1. 正常跑：注册一个用户，跑通全程，用户存了、通知发了、审计记了。
2. 只改配置：渠道 `email`→`sms`，重新拼装，通知走向变了，但**没改任何零件**，仍跑通。
3. 接错报错：故意把两个接不上的零件接一起，拼装前就报错，而不是跑起来才崩。
4. 结果稳定：同样配置、同样输入，跑两次结果完全一样。

第 2 条是本实验的核心证据。

## 工程与依赖白名单

- 本目录是独立小工程：TypeScript + 测试库（vitest + fast-check）+ 契约库（TypeBox + Ajv）。
- `allowlist.json` 登记每个依赖（依赖白名单纪律 D-006），可审计。

## 和实验③的关系

这里的零件，实验③原样复用——实验③只加一个"短信"转接件、改配置，拼出"短信注册流"，验证"不改零件也能复用"。

## 实际结论

**甜区假设为真。** 在"注册"这类结构稳定的流程上，只改配置就能改行为，不用碰装配块代码。

证据：
- 16 个测试全过（5 个属性测试验证确定性 + 5 条冒烟验证核心假设 + 6 条块级单元）
- 核心冒烟第 2 条：配置里 notifyChannel 从 email 改成 sms，没动任何块代码，通知渠道就变了
- **2026-06-26 升级**：已迁移到 `@assemflow/core` 的 `assemble()`——拼装脚本不再自带 mini-registry，真正经过引擎的 Ajv 输入/输出双契约校验、checkConfig 静态分析、异常短路。`npx tsx src/assemble.ts` 可直接跑通端到端。

迁移中暴露并补上的引擎缺口：
- 引擎 execute 缺 try-catch → 已补（块抛异常 → AssembleResult.error）
- checkConfig 静态分析只认 params、不认运行时入参 → 已补（接受可选 initialInput）

迁移中绕开但未根治的引擎缺口（记录在 docs/open-questions.md）：
- inputMap 只能"重命名"、不能构造嵌套对象或提供默认值 → 用扁平化 user-store 契约 + initialInput 初始化空数组绕开

局限（诚实标出没验证的）：
- 没测条件分支（本流是纯线性）
- 没碰真实网络/数据库（全部内存模拟）
- 没验证"演化"（块升级后配置能否接住）——这是 K-EVOL 的范围
