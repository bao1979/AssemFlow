# 复杂度搬不掉，只能搬运

> 本课目标：理解 AFP 为什么不会重蹈"积木复用"30 年失败的覆辙。
> 完整论证见 [可行性分析](../装配流编程-可行性分析.md)，本课只讲要点。

## 一条元定律

> 一个系统的本质复杂度，既不能被工具消灭，也不能被中央设计者预先决定；只能被搬运到更便宜的地方，并允许它在使用中演化。

它站在三位前人肩上：

- **Brooks《没有银弹》**：复杂度分本质的和偶然的。工具只能砍偶然复杂度（语法、样板），砍不动本质复杂度（业务本身的判断、分支、映射）。
- **Tesler 复杂度守恒**：本质复杂度是常数，只能在"用户扛"和"系统扛"之间搬，搬不掉。
- **Hayek 知识问题**：什么粒度才正确，这个知识分散在所有使用者的踩坑经验里，设计时不存在于任何人脑中。

## AFP 的本事不是消除难，是搬运难

AFP 不减少业务的本质复杂度。它做的是：**把复杂度从命令式代码，搬到声明式配置。**

这笔买卖只在"配置是承载这坨复杂度的更好介质"时才划算：

- **划算**：结构性、接线性的复杂度（像 SQL 之于循环）。
- **灾难**：算法性、条件爆炸的复杂度——一旦在配置里写算法，就退化成"一门很烂的编程语言"。

这就是为什么 AFP 有条硬边界：**算法不入配置，只留在装配块里。**

## 机制可复用，策略不可复用

软件复用 30 年失败的真相是这一句：

> **机制（mechanism）可复用，策略（policy）不可复用。**

- lodash 能全球复用，因为它是纯机制，和业务无关。
- "用户注册"维护成本高，因为它是策略，业务一变它就变。把策略当组件卖，必败。

AFP 的应对：把易变的策略从装配块里挤出去，锁进转接件（一个局部、廉价、可丢弃的地方）。装配块只留纯机制，于是它能像 lodash 一样不变、可复用。

> 这就是上一课那个结论的由来：**"用户注册"不是零件，是装配流；能复用的只有纯机制块。**
