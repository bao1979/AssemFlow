# 复用：拼第二条流

> 本课目标：同一批块、不改代码，在第二条流里复用。

## 实验③做了什么

在 `experiments/exp03-wrap-reuse/` 里，从 lodash 封装了两个块：
- `capitalize`（首字母大写）
- `deep-get`（从嵌套对象取值）

然后用**同一批块**写了两条完全不同的流：
- 流 A（`assemble-flow-a.ts`）：从用户 profile 取名字 + 大写
- 流 B（`assemble-flow-b.ts`）：从商品数据取标题 + 大写

两条流各自是一段调用同样块的 TS 代码，零修改块代码——只是传入参数不同。

**诚实标注**：当前实验③的"复用"是**块复用**（同一份块代码被两段不同的装配代码调用），而非"配置驱动复用"（还没有像实验①那样用 JSON 配置来声明第二条流）。将来引擎成熟后，第二条流只需要一份新 JSON 配置指向同一批块——但现在还没做到那步。

## 跑一下

```powershell
cd experiments/exp03-wrap-reuse
npm install
npm test
```

看到流 A 和流 B 都通过。

## 关键观察

- `capitalize` 不知道自己在处理"用户名"还是"商品标题"——它只知道"把字符串首字母大写"。
- 是**配置**告诉引擎"这次拿什么数据喂给 capitalize"。
- 同一个块被两条流复用，不改一行代码。

这就是"装配块是纯机制，可全球复用"的实际证明。

## 封装成本

从 lodash 封装一个纯函数成合规块要多少？

- 薄壳代码：1-2 行（调 lodash + 返回）
- Schema 声明：5-8 行（TypeBox 写输入/输出）
- 总计约 15 行/块

极低成本。npml 里几十万纯函数都是潜在的装配块原料。

## 反例：非纯函数封不了

实验③还故意封了 `_.now`（读系统时钟）。属性测试自动抓到它"两次调用结果不同"——不确定，不合规。

> AFP 的工程纪律（属性测试验证确定性）自动挡住了不合格的块，不靠人肉审查。
