# 亲眼看确定性

> 本课目标：验证"同配置 + 同输入 → 永远同结果"，理解"运行期零 AI"的意义。

## 跑两次，对比结果

```powershell
cd experiments/exp01-sweet-spot
npm run assemble > result1.json
npm run assemble > result2.json
```

对比两个文件——完全一样。

这不是碰巧。是因为：
- 块是纯函数（同输入同输出，已被属性测试验证）
- 拼装脚本只做"读配置、按顺序调"——没有随机、没有时钟、没有 AI
- 盐和时间戳都作为固定输入传入（不偷读系统状态）

## 为什么"确定性"值得单独讲

| 不确定的系统 | 确定的系统 |
| :--- | :--- |
| 出了 bug 很难复现 | 同输入必复现——抓 bug 容易 |
| 测试不稳定（flaky） | 测试永远稳定 |
| 审计困难（同操作不同结果） | 可审计（同配置同结果，差异必来自配置变更） |
| 回滚有风险（回到旧版不一定恢复） | 回滚可靠（旧配置 = 旧行为） |

AFP 的确定性保证来自三层：
1. 块是纯函数（属性测试守门）
2. 引擎不引入随机/时钟/AI（设计上禁止）
3. 需要随机/时间的地方，显式传入（把不确定性赶到系统边界）

## 冒烟测试里的确定性断言

每个实验的冒烟测试都有一条"跑两次结果一样"的断言。如果未来谁改了块、偷偷引入了不确定性，这条测试会立刻红——自动守门。
