# 粒度不能自上而下定义

> 本课目标：理解 AFP 为什么不去定"块该多大"，而去保证"切错了能廉价改对"。
> 完整论证见 [可行性分析](../装配流编程-可行性分析.md)，本课只讲要点。

## 别去"解"粒度问题

一个块该多大、该怎么切？由 Hayek 知识问题：**试图自上而下定义粒度标准，本身就是错的**——这是中央计划思维。

npm 的包粒度不是谁规定的，是千万人 fork、重复、磨合后**自然收敛**出来的。

所以 AFP 换一个目标函数：

> 不追求"一次切对"，追求"切错了能廉价改对"。**优化可演化性，而非前期正确性。**

## 怎么做到"改对很便宜"

- **治理关节，放开零件**：只钉死契约（输入/输出 Schema、版本、转接协议），让粒度自由浮动。正如 npm 标准化的是 `package.json` 和版本号，不是"包该多大"。
- **让重构安全**：块有严格 Schema + 测试，拆分/合并时类型和测试会接住破坏。带契约的粒度，好改。
- **切细优先**：组合是可逆的（小块能打包成复合流），分解是破坏性的（粗块拆开要开膛）。拿不准就切细。

## 演化不会停在"完美"，会停在"锁定"

lodash 粒度不再变，不是因为完美，而是改动成本太高没人敢动。一个被很多流依赖的块也一样。

应对：

- v1.0 前鼓励激进重构；v1.0 后走语义化版本，引擎支持**多版本共存**，新旧并行。
- 块是纯机制 + 强契约，版本差异在结构上是机械的，引擎**或许**能借此把配置半自动迁到新版。

> ⚠ **诚实提示**：上面这套演化方案目前是**假设，不是已证结论**。可靠演化是 AFP 最不确定的一环——语义级的破坏性变更未必能自动迁移、多版本共存可能碎片化、组合后的"特性交互"问题尚无解。这一点必须在实验中证明，而不是想当然。

## AFP 独有的一点

npm 的粒度是为"人类装配工"优化的——人怕找、怕接线，所以收敛到省人力的粒度。

而在 AFP 里，找和接线的活主要由 **LLM** 干，LLM 不怕接 200 个小块。

> 所以 **AFP 的最优粒度可能比 npm 更细**。这是一道前人问不出的开放问题。
