# 五个零件

> 本课目标：先用一条"河"建立感觉，再记住五个零件的名字。

## 用一条河来想

把一条业务流程想象成一条河：

- 河里流的**水**，是数据。
- 约束水流的**河床**，是不变的机制。
- 需要分流时筑的**堰**，是按业务需要做的适配。
- 一张**河图**，规定了这条河怎么走。
- 最后**流动的整条河**，就是跑起来的业务。

这五样，就是 AFP 的五个零件。

## 五个零件

| 零件 | 河的比喻 | 一句话 | 超小例子 |
| :--- | :--- | :--- | :--- |
| 装配块 | 河床 | 纯机制，和业务无关，到处能用 | "判断邮箱格式对不对" |
| 转接件 | 堰 | 按这单业务做的适配、分流 | "这个业务的用户字段叫 mail，转成块要的 email" |
| 配置 | 河图 | 声明式蓝图：哪些块怎么接起来 | "先校验邮箱，再存用户，最后发通知" |
| 数据 | 河水 | 流动的运行/测试数据 | 一个具体用户：`{mail: "a@b.com"}` |
| 装配流 | 整条河 | 上面四样拼出的完整业务流 | "用户注册"这条流 |

## 哪个能复用，哪个不能

这是 AFP 最关键的一点，先记住：

- **装配块能全球复用**。因为它是纯机制——"校验邮箱"这件事，全世界的业务都一样。它像 lodash。
- **转接件、配置、数据不能复用**。因为它们绑定了具体业务——每家的字段、规则、数据都不同。

所以 AFP 有个反直觉的结论：

> **"用户注册"不是一个可复用的零件，它是一条装配流**（由块 + 转接件 + 配置 + 数据拼成）。
> 能复用的只有它用到的那些纯机制块。
