# 什么时候别用 AFP

> 本课目标：诚实讲 AFP 的边界。可信度来自敢讲自己不行的地方。

## 实验②找到的崩点

我们用"订单折扣计算"做了边界实验（`experiments/exp02-boundary/`）。结论：

**AFP 在"条件密集 + 频繁变的业务规则"上失效。**

具体崩在哪：折扣规则的**条件**是函数（如"金额 ≥ 200"），函数不能放 JSON 配置——物理上塞不进。但业务想频繁改条件、不想改代码。AFP 的"只改配置就能改行为"在这里不成立。

## 适用边界线

| 在线内（用 AFP） | 在线外（别强用 AFP） |
| :--- | :--- |
| 步骤间线性或枚举路由 | 步骤结构取决于运行时数据（动态管道） |
| 策略变更 = 切换枚举值、调参数 | 策略变更 = 修改条件逻辑/算法 |
| 块可复用、与业务无关 | 条件密集的业务规则频繁变 |
| 少量规则、组合可穷举 | 多规则叠加、组合爆炸 |

## 碰到边界怎么办

AFP 纪律第 6 条：**落入"10% 区"不硬装。** 应对方式：

1. **回退到人工/对话式开发**——承认这块不是 AFP 的活，用传统代码写。
2. **把复杂判断封进一个"规则引擎块"**——块内部用 DSL 或数据驱动，外部通过输入数据改规则。复杂度没消失，但被圈在了块里、有单测兜底。
3. **引导沉淀新块**——如果这类复杂判断反复出现，把它标准化成一个可复用的块（如"通用规则引擎块"），未来类似场景直接复用。

## 一句话

> AFP 不是银弹。它解决的是"结构稳定、变化可枚举"的那 90%。剩下的 10%，承认它、圈住它、别让它污染甜区。
