# AI 介入实测提示词自测包（交付物 A · Q-003）

> 本文件是路线图 `docs/paradigm-validation-sokoban-roadmap.md` 「交付物 A」的实体。
> 用途：让 Plucker518 或任何贡献者，拿现成的提示词到各家大模型里跑一遍，产出 **Q-003（AI 产配置）** 的首批实测数据。
> **非自动验收**——结果由跑测者主观裁定并回记本文件。

---

## 这份文档解决什么

- Q-003 长期是项目最大空档：AFP 的核心叙事是「AI 产配置、引擎确定执行」，目前只证过后半段。
- 本文件把「AI 产 AFP 配置」的实测门槛拉到最低：**打开→复制→粘贴→跑→回填一张表**。
- 目标不是「统计通过率」，而是**积累模型的具体错误 pattern**——为后续 Studio 的 AI 交互设计提供依据。

**成功 / 失败同价**：模型跑不出合规产出是**有价值的信号**（说明当前上下文包不够 / 提示词写得不好 / AFP 契约对 LLM 不友好）。据此改进比"跑通"更有意义。

## 使用流程

### 单次实测的最小工作单元

跑一次 = 完成一个"轮次记录"。步骤：

1. **选一条提示词**（A / B / C 三条，见下）。
2. **打开一个新对话**（推荐 Claude Sonnet / Opus / GPT-4o / Gemini Pro，任一都行，多家更好）。
3. **先粘贴 `docs/agent-test-context.md` 全文**作为上下文。
4. **再粘贴该条提示词的"提示词正文"部分**。
5. **不追问、不引导**——只看首轮产出（本 MVP 版关注一次通过率，不看多轮修复）。
6. **判断产出**：
   - 用「人工判断观察点」清单逐项打钩。
   - 若产出是关卡文本，把它保存成 `.txt` 文件，跑 `npx tsx scripts/check-level.ts <path>` 得到 CLI 判定作为**客观补充证据**（但**最终裁定权在人**，不是 CLI）。
7. **填一条"轮次记录"**（下方模板）到本文件对应提示词的"实测记录"段。
8. **可选**：在同一条提示词上换一家模型再跑一次——横向对比更有信息量。

### 一次跑多少轮才算够？

- **首批目标**：每条提示词至少 **3 次实测**（可跨模型），共约 **9 条记录**。
- **足够动手改的信号**：连续 2 条相同错误 pattern → 说明"上下文包 / 提示词"对该类任务不合适，可以立即改进而不必再跑更多轮。
- **不追求大样本**：Q-003 要的是"AI 在 AFP 结构下的定性表现"，不是统计学显著性。

### 隐私 & 数据处理

- 实测记录里可放**关卡文本 / 模型产出的原文**——都是本项目公开产物，无隐私。
- **不要**在提示词里附带你的 API Key、真名、非本项目代码。

## 提示词 A · 加一关（指定尺寸与难度）

### 任务目标

让 LLM 产出一份**新的 Sokoban 关卡 ASCII 文本**，需要满足：
- base 4 条规则（详见上下文包）
- 发表关额外硬约束（≥2 箱、≥2 目标、开局非通关）
- 用户指定的尺寸 & 难度语义

### 上下文清单（跑测者要粘贴的内容）

- 必须先粘贴：`docs/agent-test-context.md` 全文。
- 提示词正文（下一节）。
- **不额外提供**代码 / 测试文件——检验 LLM 是否能只凭上下文包完成任务。

### 提示词正文（复制以下 fenced block 到模型对话）

```
你是一位在 AssemFlow 项目里给 Sokoban 网页小游戏设计新关卡的贡献者。
上文的《AI 实测上下文包》就是你要遵守的全部规则。

请设计一份新的 **发表关（publishable）**，要求：
1. 网格尺寸约 8×8（±2 均可），四周必须由 '#' 完全闭合。
2. 恰好 2 个箱子 '$' 与 2 个目标格 '.'。
3. 恰好 1 个玩家 '@'。
4. 开局非通关（不允许一个箱子已在目标格上）。
5. 难度：中等——玩家从开局出发**不能在 3 步内**通关，
   但也**不应设计成死锁**（应存在一条可推通关的路径，你不必给出解，但应尽力保证存在）。
6. 主题风格自选（例如「小房间」「H 形走廊」「L 形拐弯」等），
   在最后一行用 `# 主题: <一句话>` 说明。

**只输出这一份 ASCII 关卡文本**（含"# 主题:"注释行）。不要输出任何解释、代码、markdown fence。
```

### 期望产出形状

- 一份 6–10 行、6–10 列的 ASCII 网格
- 只用合法 7 种字符：`#`, `<space>`, `.`, `$`, `@`, `*`, `+`
- 末尾一行是 `# 主题: <一句话>`（这行**不属于关卡**，跑测者可手动去掉后再 `check-level`）

### 人工判断观察点

☐ 是否输出**只有关卡文本**（无 markdown fence、无解释、无代码块）
☐ 网格是否 ASCII 合法（7 种字符集）
☐ 四周是否用 `#` 完全闭合（视觉快检；CLI 复核会跑 `boundary-not-closed`）
☐ `@` 恰 1 个 / `$` 恰 2 个 / `.` 恰 2 个
☐ 开局是否非通关（视觉看：没有箱子恰好放在 `.` 位——那样就是 `*` 了，说明已通关）
☐ CLI 复核：把关卡保存成 `.txt` 后 `npx tsx experiments/exp06-sokoban/scripts/check-level.ts <path>`，是否 `exit 0` + "通过"
☐ 主题注释：是否理解"这行不是关卡的一部分"这个约束（模型常见错误：把主题写进网格）

## 提示词 B · 加一种新地形/新元素

### 任务目标

让 LLM 产出**一份提议**：包含"新地形/元素的规则"+"对应的装配块骨架"+"配置改动建议"。检验 LLM 能否理解 AFP 的三层分工（块 / 转接件 / 配置）并给出符合纪律的方案。

### 上下文清单（跑测者要粘贴的内容）

- 必须先粘贴：`docs/agent-test-context.md` 全文。
- 提示词正文（下一节）。
- **可选**：追加 `move-with-push.ts` 的完整源码（若模型要求）——为避免第一轮就"作弊看代码"，**默认不提供**、看模型自主设计能力。

### 提示词正文（复制以下 fenced block 到模型对话）

```
你是 AssemFlow 项目的贡献者，任务是给 Sokoban 网页小游戏**加一种新地形：冰面（ice）**。

**冰面语义**（这是玩家/贡献者约定，你需要遵守）：
- 冰面用字符 `~` 表示（新增到字符集）。
- 玩家踩到冰面后：本回合按下的方向会让玩家**沿该方向连续滑行**，直到撞墙 / 撞箱 / 出界 / 离开冰面。
- 冰面上的箱子被推时**也会同步滑行**，直到同样的三种终止条件。

请遵循 AFP 纪律，给出：

1. **契约层设计**：`GridState` 是否需要新增字段？如需要，写出新字段的 TypeBox schema（伪代码）。
2. **装配块设计**：给出**新装配块的名字与契约（input/output schema）**。若你认为需要一个新块，说明它做什么；若你认为可以复用 `move-with-push`，说明为什么。
3. **配置改动**：给出 `push.jsonc` 应如何修改（贴修改后的完整 jsonc，或说明"不需要改"并给出理由）。
4. **算法归属**：这条冰面滑行算法应该住在**块内**还是**配置里**？为什么？（提示：AFP 硬边界"算法不入配置"）
5. **`@paradigm` 标记**：这个改动是否会引入非 AFP 范式？若会，标记在哪一层、理由是什么？

输出请分成 5 个小节，每节 3-8 行，语言直接。**不要**输出完整 TS 代码——只写契约 schema、命名、语义。
```

### 期望产出形状

- 5 段小结构化产出（对应 5 问）
- 装配块名称 = kebab-case（例如 `move-with-push-and-slide` 或 `ice-slide`）
- 契约 schema 用 TypeBox 风格伪代码
- **对硬边界的态度**：模型是否**主动**说"算法留块内"、不塞进配置？

### 人工判断观察点

☐ 是否理解 AFP 三层分工（**不把滑行算法写进 jsonc**）
☐ 是否给出**结构化产出**（5 段清晰）而不是长篇散文
☐ 契约设计是否合理（例如：新地形要不要加进 GridState.floors[]？还是 tiles?）
☐ 装配块命名是否符合 kebab-case + 语义清晰
☐ 是否**正确回答**"算法归属块内"这一硬边界问题（这是 AFP 核心纪律测验）
☐ 是否识别出可能的 `@paradigm` 触发点（例如"滑行需要循环"是否要 `@paradigm NON-AFP: loop-in-block`）
☐ 是否**不合理地建议**改引擎（例如"给引擎加 loop step"——这是 Q-027 已决定不做的方向）

**首轮强项（可写进汇总的正面证据）**：

- 硬边界 5/5 纪律测验通过：拒绝把算法塞进 jsonc、拒绝改引擎、拒绝推 loop step、拒绝继承、`inputMap` 只做字段重命名
- 拒绝复用 `move-with-push` 的理由准确："契约不含循环终止语义"抓住要害
- `push.jsonc` 改动极度克制：只换 block name，未加 step / 分支 / 计算
- 主动指出"数据加载层的纯机械映射，不是算法"——**AFP 三层分工意识扎实**（本次实测最亮点）
- 追问后能**主动修正立场**（从"不合成"改到"必须合成"），说明它有反思能力

**新观察到的稳定错误 pattern**（记入项目级观察，供后续跨模型对比参考）：

1. **契约完备性意识局部化**：模型能补答被追问的字段（字符集），但**不会主动扫描其它下游影响**。追问回复里只提到 `check.ts` 的 `invalid-char` 规则要改，**遗漏另外三条规则**也要同步改：
   - `player-count`：玩家计数要加 `}` 和 `[`
   - `box-goal-imbalance`：boxes 计数要加 `{` 和 `]`；goals 计数要加 `%`、`]`、`[`
   - `boundary-not-closed`：flood-fill 的"是墙判定"要把 `~`、`%`、`{`、`}`、`]`、`[` 都视为非墙
   模型泛泛地说了句"解析器同步升级"——**是"局部完备"意识，缺"全局扫遍影响面"的自觉**。对目标用户是"非程序员"的 Studio 而言，这个盲区不解决，AI 直接产的产出永远要跑几次"你还漏了 X"的追问循环。

2. **无产品级考量（字符选择只顾合法性、不顾可读性）**：模型追问后新增 5 个合成字符 `~ % { } ] [`（共 12 字符集），但字符选择**产品级瑕疵明显**：
   - `{` 和 `}` 视觉是"左右花括号一对"，等宽字体下手写关卡极易看错
   - `[` 和 `]` 同样问题
   - `{` = ice+box、`]` = ice+goal+box——**同是"含 box"**，字符**毫无视觉关联**
   - 对比传统 `*`（= `.` + `$`）、`+`（= `.` + `@`）——传统字符**有"叠加感"**，模型选的新字符**没有**
   这是 AI 在"设计工作"和"程序员工作"之间的差距——**只优化"字符合法唯一"，不问"用户看这份产出会不会看错"**。

**次要观察（不构成 pattern，但值得记）**：

- 组合完备性对（6 个字符覆盖所有可能组合），但**没显式讲**"箱与玩家互斥，故省略 ice+box+player / ice+goal+box+player"的排除理由
- `@paradigm` 那节结论对但推理链薄——没显式说"块内部实现（for/while/递归）不入 AFP 数据流承诺范围"这个核心理由；只说"配置里没循环所以没事"是浅层解释

**模型原始产出 · 首轮**（对提示词 B 正文的回复）：

模型的 5 段结构化产出：

- **§ 1 契约层设计**：新增 `ice: Position[]` 字段（类比 walls/goals 静态地形）；TypeBox 伪代码 `Type.Array(Type.Object({x: Type.Number(), y: Type.Number()}))`；`~` 不与其它字符叠加（首轮判断）
- **§ 2 装配块设计**：必须新增 `move-with-slide`，不复用 `move-with-push`；契约 `{grid, direction} → {nextGrid}`；块内循环实现滑行
- **§ 3 配置改动**：`push.jsonc` step1 的 `block` 从 `move-with-push` 替换为 `move-with-slide`，其余不变
- **§ 4 算法归属**：滑动算法必须住块内，绝不入配置（`while` 循环 + 多步条件 = 算法）
- **§ 5 `@paradigm` 标记**：本次改动不引入非 AFP 范式；理由"机制封块内、配置只表达静态拓扑、GridState 加字段是合法契约扩展、未引入继承"

**追问及模型回复**（问题：`~` 位置同时是目标格/箱子/玩家时字符如何表示）：

模型反转首轮"不合成"立场，给出"必须合成"方案：

- **核心原则**："一个格子一个字符"是 Sokoban ASCII 铁律；合成是第一公民（对齐传统 `*` / `+`）
- **合法字符集扩至 12 个**（原 7 + 新 5 合成 + 1 基础 `~`）：

| 字符 | 含义 | 网格叠加状态 |
| :-- | :-- | :-- |
| `~` | 冰面（空） | 仅 ice |
| `%` | 冰面 + 目标 | ice + goal |
| `{` | 冰面 + 箱子 | ice + box |
| `}` | 冰面 + 玩家 | ice + player |
| `]` | 冰面 + 目标 + 箱子 | ice + goal + box |
| `[` | 冰面 + 目标 + 玩家 | ice + goal + player |

- **解析器逻辑**：逐字符映射到 `walls/goals/boxes/player/ice` 数组，映射规则完整枚举 12 字符
- **对 `checkLevel.ts` 的后续影响**：模型只显式提到 `invalid-char` 规则要扩到 12 字符集（**遗漏另外三条规则**——见上文"错误 pattern 1"）
- **拒绝"不合成"方案的理由**："`~$` 两字符表示一格会破坏坐标对齐"——理由正确，但没提"多字符+分隔符"等替代方案作对比

- 单次实测（模型 A），未跨模型验证。**两条错误 pattern 是否稳定**需要至少再跑 1-2 家其它模型（同一版 `agent-test-context.md`、同一条提示词 B、不给追问）确认。
- 建议下一步：不再追问 B、改跑 C（修 bug 关卡，最短）→ A（加一关，Studio 未来核心用例）→ 换家跑 B（跨模型对比）。
- 若两条 pattern 跨 3 家模型稳定复现 → 触发一次 `docs/agent-test-context.md` 改进（在 § 11 补两条 pattern 提醒）。当前**不改**上下文包，保持实验公正性。

## FAQ

**Q：为什么不给"通过率 X% 视为 AFP 成功"这样的自动判据？**
A：Q-003 的核心不是"AI 每次都能对"，而是"AI 在 AFP 结构下的错误模式**是否稳定、是否可通过 prompt/skill 消除**"。前者是产品指标（等 Studio 有了再说），后者是范式判据。当前阶段只积材料，不下判决。

**Q：一次跑不出结果就否定 AFP 吗？**
A：不。**单次失败**只说明"这一条提示词 + 这一个模型 + 这一版上下文包"这个组合下失败——用来指导上下文包 / 提示词 / 甚至 AFP 本身的改进方向，不是范式的死刑判决。

**Q：能不能把这份自测包做成自动脚本？**
A：可以，但**先跑够 3–5 条手工记录**再脚本化。理由：脚本会自动汇总掉"模型产出的具体错误措辞"，而**这些原文正是当前阶段最有价值的信号**。等错误 pattern 稳定下来再脚本化不迟。若确实想早做，加一个 `scripts/agent-runner.ts` 用 API 批量跑即可，不必等本文件收工。

**Q：贡献自己的实测记录？**
A：直接编辑本文件对应提示词的"实测记录"段，追加一条轮次记录，PR 提交即可。这是 `CONTRIBUTING.md` 的一等贡献路径。
