## Q-001 · 演化可靠性未经证明 · 🔴

**问题**：装配块发生 breaking change（schema 改变、行为改变）时，能否可靠地把已有配置/转接件迁移到新版？

**为什么关键**：这是 AFP 最薄弱的一环。如果块一升级就全网炸（配置失效、转接件失效），"全球共享装配块"的愿景直接塌。

**当前状态**：纯理论假设。已在 afp-core.md 标注"待验证"。

**验证思路**：扩展实验③——故意让一个块（如 capitalize）发生 schema 变更（v1→v2），看引擎/配置/转接件是否能接住，以及代价多大。

**判据**：
- 自动 codemod 在简单 schema diff 上能否成立（加字段、改名）
- 多版本共存能否避免碎片化
- "特性交互"（两个块各自升级都对、组合起来坏）能否被静态分析挡住

**依赖**：无；可独立启动。

## Q-002 · 配置驱动复用没真正做到 · 🔴

**问题**：实验③声称"复用"，但实际是两段独立 TS 代码各调同一批块，**不是**"一份新 JSON 配置就拼出第二条流"。

**为什么关键**：AFP 的核心叙事是"配置驱动装配"。如果配置只能驱动单条流、不能驱动复用，那"块全球复用"的故事就缺一道关键证据。

**当前状态**：教程 05 课已诚实标注此局限。但局限本身没被解决。

**验证思路**：基于实验③的两个 lodash 块，写两份 JSON 配置（而非 TS 代码），用引擎的 `assemble` 拼出流 A 和流 B。

**判据**：
- 两份配置都用 `@assemflow/core` 装配，零额外 TS 胶水
- 配置变更（如改路径）不需要改任何块代码或装配代码

**依赖**：先解决 Q-005（引擎端到端接管真实流）。

## Q-003 · AI 产配置完全没验证 · 🔴

**问题**：AFP 的核心叙事是"AI 产配置、引擎确定执行"。目前只证明了"人产配置、引擎确定执行"——AI 那一半零证据。

**为什么关键**：如果 LLM 实际产不出合规配置（频繁报错、违反静态可枚举红线、引用不存在的块），整套"AI 时代编程范式"的卖点就只是宣传。

**当前状态**：未启动。

**验证思路**：
1. 准备：现有块清单 + 一句业务需求描述
2. 给 LLM（如 Claude）这两份输入，要它产出一份配置
3. 用 `assemflow check` 校验该配置——看零修改下能不能过
4. 跑多次/多需求，统计通过率与常见错误

**判据**：
- 一次通过率 ≥ 多少算可用？（这个阈值本身需要先讨论）
- 常见错误类型是什么——能否通过 prompt/skill 设计消除？

**依赖**：Q-005（先有真实引擎可校验）+ 当前块清单足够覆盖一个真实场景。

## Q-004 · 高阶函数 / 副作用库的封装未探索 · 🟡

**问题**：实验③只封装了 lodash 的简单签名（单参/多参纯函数）。带回调的高阶函数（如 `_.map(arr, fn)`）、有副作用的库（fs/http）封装难度未知。

**为什么重要**：决定"npm 整个生态能否做装配块原料库"的覆盖率。如果只能封装 10% 的 npm 函数，规模化路线就缩水。

**当前状态**：未启动。

**验证思路**：
- 高阶函数：把 `fn` 参数降级为"操作描述数据"（让块内部分发），守住"算法不入配置"
- 副作用库：要么"封掉副作用、把它变成纯数据变换"（如 fs.readFile → 输入文件内容字符串），要么明确归为"运行环境块"（执行边界）

**判据**：能否给出一套通用封装模式，让封装成本仍然 ≤ 20 行/块。

**依赖**：无。

## Q-005 · 引擎尚未端到端接管真实业务流 · 🟡

**问题**：当前 `@assemflow/core` 只在 `engine/tests/` 里跑过 mock 块。实验①还是用手写脚本，没接引擎。"通用引擎能跑真实业务流"还没被证明。

**为什么重要**：是 Q-002/Q-003 的前置条件。引擎没接管真实流，配置驱动复用和 AI 产配置都没意义。

**当前状态**：引擎 MVP 已实现（13 测试），但未在实验上端到端。

**验证思路**：把实验①的 `assembleAndRun` 改写为调 `@assemflow/core`：用引擎的 `BlockRegistry` 注册 5 个块，用引擎的 `assemble` 替代手写的 for 循环。

**判据**：
- 实验①的 16 测试在迁移后全部通过
- 不再需要"注册流专属的字段映射胶水"，全靠 inputMap

**依赖**：可能需要扩展引擎（如更灵活的 inputMap 表达）。

## Q-010 · 安全：恶意装配块的防护边界 · 🟡

**问题**：之前讨论过——块越多人用，一个被注入恶意算法的块危害越大。AFP 如何防？目前没具体方案。

**为什么重要**：与"全球共享装配块"愿景直接冲突——共享越广，攻击面越大。

**当前状态**：仅 afp-core.md 提到"依赖白名单"原则，但白名单只能挡"未登记的依赖"，挡不住"白名单内某个包被注入恶意代码"。

**验证思路**：
- 短期：实验①已落地 allowlist.json（最小载体）
- 中期：研究"内容寻址（hash-pinned）依赖"、"签名块清单"等机制
- 长期：与 npm 已有的供应链防护（如 Sigstore）对接

**依赖**：无；可随时启动研究。
