# 待解决与待验证清单

> 这是项目所有未结议题的登记册——避免它们散落在对话里、被遗忘。
> state.json 里的 `openQuestions` / `knownIssues` 是摘要，本文件是详情。

## 分级

- **🔴 关键**：直接决定 AFP 能否成立。这些未解前，项目结论只能是"看似可行"，不能升级。
- **🟡 重要**：影响范式完整性与外部可信度，但不会单独击穿核心。
- **🟢 一般**：体验性、产品化、扩展性议题。

## 状态

- `open`：未解
- `in_progress`：进行中
- `resolved`：已解（含证据链接），保留一段时间后归档
- `archived`：已归档

## 可能的结局（resolution）

每条问题最终不一定要"解决"，可以是：

- **`verified`**：通过实验/证据证明假设成立
- **`falsified`**：通过实验证明假设不成立，需调整范式
- **`avoided`**：通过设计绕开（明确"AFP 不处理这类场景"，划在边界外。这是合理结局，不是失败）
- **`deferred`**：当前不解、未来再说，必须附"何时重审"的触发条件

> "绕过 / 避免"也是合法解。一个范式知道自己不做什么，比假装什么都能做更可信。

## Q-001 · 演化可靠性未经证明 · 🔴

**问题**：装配块发生 breaking change（schema 改变、行为改变）时，能否可靠地把已有配置/转接件迁移到新版？

**为什么关键**：这是 AFP 最薄弱的一环。如果块一升级就全网炸（配置失效、转接件失效），"全球共享装配块"的愿景直接塌。

**当前状态**：纯理论假设。已在 afp-core.md 标注"待验证"。

**验证思路**：扩展实验③——故意让一个块（如 capitalize）发生 schema 变更（v1→v2），看引擎/配置/转接件是否能接住，以及代价多大。

**判据**：
- 自动 codemod 在简单 schema diff 上能否成立（加字段、改名）
- 多版本共存能否避免碎片化
- "特性交互"（两个块各自升级都对、组合起来坏）能否被静态分析挡住

**依赖**：无；可独立启动。

## Q-002 · 配置驱动复用没真正做到 · 🔴

**问题**：实验③声称"复用"，但实际是两段独立 TS 代码各调同一批块，**不是**"一份新 JSON 配置就拼出第二条流"。

**为什么关键**：AFP 的核心叙事是"配置驱动装配"。如果配置只能驱动单条流、不能驱动复用，那"块全球复用"的故事就缺一道关键证据。

**当前状态**：教程 05 课已诚实标注此局限。但局限本身没被解决。

**验证思路**：基于实验③的两个 lodash 块，写两份 JSON 配置（而非 TS 代码），用引擎的 `assemble` 拼出流 A 和流 B。

**判据**：
- 两份配置都用 `@assemflow/core` 装配，零额外 TS 胶水
- 配置变更（如改路径）不需要改任何块代码或装配代码

**依赖**：先解决 Q-005（引擎端到端接管真实流）。

## Q-003 · AI 产配置完全没验证 · 🔴

**问题**：AFP 的核心叙事是"AI 产配置、引擎确定执行"。目前只证明了"人产配置、引擎确定执行"——AI 那一半零证据。

**为什么关键**：如果 LLM 实际产不出合规配置（频繁报错、违反静态可枚举红线、引用不存在的块），整套"AI 时代编程范式"的卖点就只是宣传。

**当前状态**：未启动。

**验证思路**：

**判据**：
- 一次通过率 ≥ 多少算可用？（这个阈值本身需要先讨论）

**依赖**：Q-005（先有真实引擎可校验）+ 当前块清单足够覆盖一个真实场景。

## Q-004 · 高阶函数 / 副作用库的封装未探索 · 🟡

**问题**：实验③只封装了 lodash 的简单签名（单参/多参纯函数）。带回调的高阶函数（如 `_.map(arr, fn)`）、有副作用的库（fs/http）封装难度未知。

**为什么重要**：决定"npm 整个生态能否做装配块原料库"的覆盖率。如果只能封装 10% 的 npm 函数，规模化路线就缩水。

**当前状态**：未启动。

**验证思路**：
- 高阶函数：把 `fn` 参数降级为"操作描述数据"（让块内部分发），守住"算法不入配置"
- 副作用库：要么"封掉副作用、把它变成纯数据变换"（如 fs.readFile → 输入文件内容字符串），要么明确归为"运行环境块"（执行边界）

**判据**：能否给出一套通用封装模式，让封装成本仍然 ≤ 20 行/块。

**依赖**：无。

## Q-011 · 转接件是否会成为新的技术债中心 · 🔴

## Q-012 · 真实副作用与交付语义未定义 · 🔴

**判据**：能否给出一套清晰的"块分类 + 副作用语义"，且引擎能在配置层表达重试/幂等/补偿。

## Q-014 · 装配块的身份/打包/发布规范未定义 · 🟡

**问题**：一个块如何被唯一标识？版本怎么声明？schema 制品怎么携带？发布给谁、引擎怎么发现？目前散落。

**为什么重要**：你已经定调"npm 是原料库"，但"AFP 块作为一等资产"的包规范还没明确。后续共享、检索、签名、白名单、影响面分析都建立在此之上。

**当前状态**：实验①/③临时用文件路径引用，没有正式规范。

**验证思路**：定义最小块清单格式（block manifest）：
- `name`（唯一标识，包含命名空间）
- `version`（semver）
- `inputSchema` / `outputSchema`（JSON Schema 制品路径）
- `entry`（运行时入口）
- `hash`（内容寻址，防篡改）
- `dependencies`（runtime 与 dev 分开，白名单兼容）

**判据**：一份格式能同时被引擎的 `--blocks` 参数、`assemflow check`、未来的注册表 fetch 使用，无需重新设计。

**依赖**：无；独立可推进。

## Q-015 · 多语言 / 多运行时块如何接入 · 🟡

**问题**：参考实现是 TypeScript。若块来自 Python / Rust / WASM / HTTP 服务，AFP 还成立吗？

**为什么重要**：决定 AssemFlow 是"TS 世界内的好方法"还是"更广义的软件装配方法"。当前叙事有"全球共享块"味道，若块只能是 TS，外延比文档暗示小很多。

**当前状态**：未启动。

**验证思路**：选一种最小跨边界形式先验证：
- **HTTP block**：契约通过 JSON Schema、调用通过 HTTP、确定性依赖远端实现（外部副作用边界）

**依赖**：Q-014（块身份规范，跨语言需要更标准化的清单）。

## Q-017 · 可观测性与调试证据链未定义 · 🟡

**问题**：真实流跑起来后，开发者如何看每一步输入、输出、耗时、失败点、契约校验结果？

**为什么重要**：AFP 号称"比 AI 直接写代码更可治理"。若调试体验比普通脚本还差，承诺站不住。

**当前状态**：当前 `assemble` 只返回 `{success, context, error}`。没有 step trace、contract result、context snapshot。

**验证思路**：定义最小运行报告格式：

**判据**：失败时能从报告里 3 秒定位"在哪一步、哪个字段、为什么"。

## Q-024 · inputMap 表达力受限 · 🟡

**问题**：引擎当前的 `inputMap` 只支持"字段重命名"（`{blockField: contextKey}`）。三件常见事都做不到：
1. 构造嵌套对象（例如 `{newUser.email: "email"}`）
2. 默认值（例如某个字段在上下文不存在时回退到 `[]` 或常量）
3. 字面常量（例如把 `action` 写死为 `"register"`）

**为什么重要**：表达力不足时，要么"扁平化块契约把业务字段塞进块"，要么"在 initialInput / params 里塞 hack 种子"——两种都把 ergonomic 缺口推给业务侧，违反"配置即接线"的纪律。

**状态**：`open`

**当前已绕开方式**（实验①迁移期间）：
- `user-store` 扁平化（输入从 `{users, newUser: User}` 改成 `{users, email, passwordHash}`）
- `users/sent/logs` 在 `initialInput` 里注入 `[]` 作种子
- `auditAction` 写死在 `params` 里当字面常量

**验证思路**：扩展 `StepConfig.inputMap` 的值语法。最小可行方案：
```ts
inputMap?: Record<string, string | { from?: string; default?: unknown; value?: unknown }>
```
- `string`：当前行为，重命名
- `{ from }`：等价于 `string`
- `{ from, default }`：找不到时用默认值
- `{ value }`：字面常量
- 嵌套对象构造：用点号路径（`"newUser.email": "email"`）或单独的 `compose` 字段

**判据**：实验①能撤掉 user-store 扁平化、撤掉 initialInput 种子、把 `auditAction` 用字面常量写在 inputMap 里——配置变得更显式、块契约更纯。

**依赖**：无；可独立推进。

## Q-025 · 流签名（FlowConfig 的入参契约）未定义 · 🟡

**问题**：`assemble(config, registry, initialInput)` 的 `initialInput` 是裸 `Record<string, unknown>`，没有 schema、没有类型校验。`checkConfig` 现在能识别 initialInput 的"字段存在性"，但不知道字段类型——只能等到运行时 Ajv 在第一个用它的块入口处兜底。

**为什么重要**：流也应该有自己的"对外契约"——和块契约同等公民。否则 AFP 的"强契约"承诺只兑现到块级，流级是空的。

**状态**：`open`

**验证思路**：给 `FlowConfig` 加可选字段：
```ts
inputs?: TSchema; // 流自身的入参 schema，用 TypeBox 写
```
`assemble` 在调用前用 Ajv 校验 `initialInput` 是否符合 `flow.inputs`。`checkConfig` 用 `inputs` 的 properties 喂给 available（取代当前从裸 `Record` 取 keys）。

**判据**：实验①的 `RegisterInput` 能从 TS 接口升级成配置里声明的 `inputs` schema，类型不匹配时装配前就报错。

**依赖**：Q-024（一起做更顺手）。

## Q-026 · K-STATE · 长寿命有状态对象的状态演化 · 🔴

**问题**：现有"流是一次性数据变换"的隐含假设在长寿命对象上扛不扛得住？订单状态、UI 组件、游戏角色等"持续存在的对象"的状态演化能否用 AFP 干净地表达？

**为什么关键**：状态机是任何非批处理程序的核心。AFP 不能描述状态机就只能做后端业务流，进不了 UI/游戏/事件驱动等主战场——直接关系到"范式适用域"的边界。

**状态**：`open`

**验证思路**：做一个最小状态机示例（红绿灯 / 自动售货机），试两种方案：
- **A. 调用方持久化**：当前状态作 `initialInput` 进，新状态作 output 出，调用方在外存储；引擎不变
- **B. 引擎承载状态**：给 `BlockDef` 加 `state` 字段或新增"有状态块"类型；引擎自动读/写状态快照

**判据**：在不引入"全局可变状态"的前提下，哪种方案能让配置更可读、AI agent 更容易产/改、调试更容易。诚实记录"AFP 范式在这一形态下别扭的地方"。

**依赖**：无；可独立启动。归属 paradigm-validation-sokoban MVP-0。

## Q-027 · K-LOOP · 循环 / 事件驱动的执行语义 · 🔴

**问题**：当循环次数依赖运行时数据（游戏帧循环、事件队列消费、轮询、批处理）时，AFP 该怎么表达？现有红线"动态边界循环必须封进块"违背了"配置是接线"的承诺——把控制流推进块内部 = 在 sub-graph 里造一门很烂的编程语言。

**为什么关键**：AFP 当前是 DAG，不是图灵完备控制流。任何需要主循环的程序在这里都憋住。

**状态**：`open`

**验证思路**：**不单设独立实验**——已决定并入 paradigm-validation-sokoban 的 MVP-1（走路），spec `.kiro/specs/sokoban-mvp-1-walk/`。Sokoban 的"按键→回合→渲染"循环本身就是 K-LOOP 的真实场景，比脱离场景的轮询累加器更有意义。在 MVP-1 中对比两方案：
- **A. 主循环在外部**：调用方每次按键调一次 `assemble`，引擎保持纯 DAG
- **B. 引擎支持 loop step**：`StepConfig.type = "loop"`，配置声明终止条件

**判据**：两方案在"配置可读性"、"AI agent 是否容易生成正确配置"、"`check` 静态分析是否仍成立"三维度的得失。结论写进 exp06 的 MVP-1 报告。

**依赖**：Q-026 K-STATE（状态先定型）。归属 paradigm-validation-sokoban MVP-1。

## Q-028 · 范式验证终极标的：Sokoban · 🔴

**问题**：AFP 在"网格回合制游戏"这一形态上能走多远？纯 AFP / 混合范式（AFP + reducer / 状态机 / ECS）哪个能让 AI agent 介入开发最高效？

**为什么关键**：项目目标已升级（D-013）——不只验证 AFP 是否成立，而是寻找"AI 介入模式下的更优编程范式"。游戏是载体，真正产出是一套"人 + AI agent 协作开发"的工程范式 + AFP 适用域清单。Sokoban 是最纯净标的（每关一份 ASCII 数据 = AFP "AI 产配置"的天然舞台）。

**状态**：`open`

**验证思路**：拆成一条 MVP 链。全局路线图见 `docs/paradigm-validation-sokoban-roadmap.md`；每个 MVP 是独立 spec：
- **MVP-0**：K-STATE 红绿灯状态机（= Q-026）→ `.kiro/specs/sokoban-mvp-0-k-state/`
- **MVP-1**：走路 + 渲染（含 K-LOOP = Q-027）→ `.kiro/specs/sokoban-mvp-1-walk/`
- **MVP-2**：推箱子 + 胜利判定 → `.kiro/specs/sokoban-mvp-2-push/`
- **MVP-3**：5 关 ASCII 关卡集 + 静态校验 → `.kiro/specs/sokoban-mvp-3-levels/`
- **MVP-4**：maxMoves 配置开关 + 撤销 → `.kiro/specs/sokoban-mvp-4-tuning/`

引擎参照 Godot 的词汇（Scene/Resource/Input 映射），但拒绝其架构（帧 tick / Node 继承 / Signal）。

**判据**：
1. 加新关卡 = 加纯数据，引擎零改动（MVP-3）
2. 一个配置值（maxMoves）改了行为就变，块零改动（MVP-4）
3. 非 AFP 范式处全部带 `@paradigm` 标记，最终汇成 `docs/paradigm-comparison.md` 适用域清单
4. AI 介入由提示词自测包（`docs/agent-test-prompts.md`）支撑，Plucker518 自行到大模型实测裁定，不做自动验收

**依赖**：Q-026、Q-027（即 MVP-0、MVP-1）。**不要**等 Q-001 / Q-002 / Q-003——本 spec 本身就是复合验证。
