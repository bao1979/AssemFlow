## 命名与术语体系

| 项 | 名称 |
| :--- | :--- |
| 范式中文名 | **装配流编程** |
| 范式英文全称 | **Assembly Flow Programming（AFP）** |
| 参考实现引擎/产品昵称 | **AssemFlow** |

### 一句话定义

> **装配流编程（AFP）**——一种 AI 时代的编程范式：用可复用的"装配块"，经"转接件"按"配置"驱动、对"数据"操作，装配成业务的"装配流"；**AI 负责发现与配置，引擎负责确定性执行**。AssemFlow 是其参考实现引擎。

**范式定位（与近邻划界）**：AFP 属于**声明式 + 生成式编程在 LLM 时代的形态**（不宣称"元编程的下一代"——那是过誉，易被专家判为话术）。它的产物核心不是代码，而是 **「关于如何生成代码的确定性蓝图」**；代码只是蓝图的派生物。由此与两类近邻彻底划界：

- 与**低代码**划界：低代码"隐藏复杂度"，AFP"显式化复杂度并交给图去管"。
- 与 **vibe coding** 划界：vibe coding"让 AI 猜"，AFP"让 AI 填图"。

> 一句定位：**AssemFlow 不是"AI 编程助手"，而是"业务蓝图的 CAD 工具 + 确定性编译器"。**

### 五元构件（核心术语，以江河隐喻锚定）

| 构件 | 英文 | 江河隐喻 | 理论身份 | 变化速率 | 谁来写/维护 |
| :--- | :--- | :--- | :--- | :--- | :--- |
| **装配块** | Block | 河床、河堤 | 纯机制（mechanism），不变最小组件 | 几乎不变（仅 bug/升级） | 全球共享，像 lodash |
| **转接件** | Adapter | 都江堰（按需分流） | 业务适配 / 防腐层 | 随业务变 | 各业务自己写 |
| **配置** | Config | 河图（接线蓝图） | 策略声明 / 装配接线 | 经常变 | 设计期由 AI 产、人审 |
| **数据** | Data | 河水 | 运行时数据 / 测试数据 | 时刻在变 | 业务方提供 |
| **装配流** | Flow | 流动的河 | 装配出的完整业务流 | —（由上四者组合而成） | 配置定义 |

> 术语桥接：本文档"维度分析"部分中出现的"原子流"一词，均指本节定义的 **装配块（Block）**；定稿术语以本节为准。

### 核心命题（待验证，已含愿景修正）

通过 GitHub 这样的开放平台，让全球共享 **「装配块 + 转接模式 + 配置模板」**（注意：**不是**"开箱即用的业务流"——见「由理论推出的愿景修正」一节），把 AI 的工作从"写代码"降到"管配置"，并由引擎确定性执行，实现软件资产的可复现、可审计装配。

## 理论基础（核心地基）

本节是 AssemFlow 全部设计取舍的地基。它回答一个问题：**装配流编程凭什么不会重蹈"积木复用"30 年失败的覆辙？**

### 元定律：复杂度与知识不可消灭，只能搬运与演化

> **一个系统的本质复杂度与"什么粒度才正确"的知识，既不能被工具消灭，也不能被中央设计者预先决定；只能被搬运到更便宜的地方、并允许其在使用中演化。**

这条元定律站在三位前人肩上：

- **Brooks《没有银弹》**：复杂度分**本质（essential）**与**偶然（accidental）**。工具只能砍偶然复杂度（语法、样板），砍不动本质复杂度（业务本身的判断、分支、映射）。
- **Tesler 复杂度守恒定律**：本质复杂度是常数，只能在"用户扛"与"系统扛"之间搬运，搬不掉。
- **Hayek 知识问题**：最优粒度的知识分散在所有使用者的踩坑经验里，设计时不存在于任何中央设计者脑中——正如中央计划者定不出正确价格。

→ 推论：AFP 的本事**不是"消除"写代码的难，而是"搬运难 + 放开演化"**。下面三条原则全部由此派生。

### 原则一：机制/策略二分（解"维护成本谁扛"）

软件复用 30 年失败的真相：**机制（mechanism）可复用，策略（policy）不可复用。**

- lodash 能全球复用，因为它是纯机制——与业务无关。
- "用户注册"维护成本高，因为它是策略——业务一变它就变。历史上把策略当组件卖，必败。

AFP 的五元构件正是**按"机制/策略 + 变化速率"分层**，把易变性从装配块里挤出去：
- **装配块 = 纯机制**，因此可以像 lodash 一样不变、可全球复用。
- **业务的善变被锁进"转接件（都江堰）"**——它只做"按需分流"，不改河床、不改河水。

理论支撑：**Parnas（1972）信息隐藏**（围绕"可能变化的东西"分解模块）、**Cockburn 六边形架构 / 端口-适配器**（稳定内核 + 可替换适配器）、**Stewart Brand 剪切层理论**（系统按变化速率分层）。

→ "维护成本谁扛"的问题因此消解：成本被搬到"转接件"这个局部、廉价、可丢弃的地方。代价是必须接受——**"用户注册"不是装配块，它是装配流**（由装配块 + 转接件 + 配置 + 数据拼成）。

**补充：转接件的二阶复用 / 收敛性。** 把易变性赶进转接件后，转接件自己会繁殖、重复——熵从河床搬到了闸口，并不自动收敛。解法**不是**"让转接件继承转接件"（继承=脆弱基类，会把刚甩掉的耦合请回来），而是：

> **持续把转接件里反复出现的"字段映射/类型转换"挖出来、升级成装配块（它们本就是纯机制）。转接件应越缩越小，最后只剩不可再约简的那点业务决定。**

转接件之间只用**组合/模板化**、不用继承；并对转接件套用「治理关节，放开零件」原则的同一套重复检测回路。否则只是把"代码复制"换成了"配置复制"，成本没真正收敛。

### 原则二：配置是"可被机器导航的债"（解"配置技术债"）

诚实面对：由元定律，**AFP 不减少业务的本质复杂度，只是把它从命令式代码搬到声明式配置**。这笔买卖只在"配置是承载这坨复杂度的更好介质"时才划算。

分界清晰：
- **划算**：结构性、表格性、接线性复杂度（如 SQL 之于循环、Terraform 之于手调 API）。
- **灾难**：算法性、条件爆炸复杂度——一旦在 YAML 里写算法，立刻退化成"一门很烂的编程语言"（Ansible/Helm 的 YAML 地狱）。

由此得到一条与机制/策略二分同源的**硬边界**：
> **配置只准表达"结构、接线、声明式策略"；算法必须留在装配块（代码）里。**

而"分层配置"（长江—支流）解决局部性，但**不解决影响面**。真正的杀手锏是配置相对代码的结构性优势：
> **配置天生是一张图，可被机器分析。** where-used、死配置检测、改一处的影响波及、类型校验，都能从配置结构直接算出——这比从代码里做静态分析容易得多。

→ "配置债"的解法不是"别让它膨胀"（膨胀是复杂度守恒的必然），而是：**从第一天就把依赖图谱 / 影响面分析当作引擎的一等公民。代码债看不见，配置债看得见——这是 AFP 该兑现的优势，而非弱点。**

**硬边界精化：让"配置即图"始终保持可计算。** "算法不入配置"要再钉死一层——配置图必须**静态可枚举**：

> - **允许留在配置**：对**已声明的有限域谓词**（布尔/枚举）做分支、对**带类型的集合**做有界 `for_each`。这类分支是 N 条有标签的边，影响面"乘以 N"（有限可算），不爆炸。
> - **必须封进装配块（当黑盒处理）**：**计算出来的 / 依赖运行时数据的谓词**、深层嵌套条件、动态边界循环。它们一旦出现就让那段子图不可静态分析。

"VIP 走 A 否则走 B"这种对枚举的简单路由属于前者，留在配置（业务想看见、想改）；复杂判定属于后者，入块由单测覆盖。"静态可枚举留配置、动态计算入块"是引擎能做影响面分析的红线。

### 原则三：治理关节，放开零件（解"粒度无法自上而下定义"）

由 Hayek 知识问题：**别去"解"粒度问题——试图自上而下定义粒度标准，本身就是错误的范式（中央计划）。** npm 的包粒度不是 TC39 定的，是千万人 fork、重复后自然收敛的。正确目标函数是换一个：

> **不要追求"一次切对"，要追求"切错了能廉价改对"。优化可演化性，而非前期正确性。**

四条可操作策略：
1. **治理关节，放开零件**：只钉死**契约**（I/O Schema、版本、转接协议），让粒度自由浮动——正如 npm 标准化的是 `package.json`/semver，不是"包该多大"。
2. **让粒度重构变安全变便宜**：装配块有严格 Schema + 测试，拆分/合并时类型系统与测试会接住破坏。带契约的粒度好改，代码粒度难改。
3. **播种—观测—收敛，把 fork 当信号**：建反馈回路（谁被复用、谁被 fork/改写），让数据告诉你哪里该拆该并。你建的是"回路"，不是"答案"。
4. **切细优先**：组合是可逆的（小块可用具名配置打包成"复合装配流"），分解是破坏性的（粗块拆开必须开膛）。故拿不准时切细；细 + 复合命名 + 分层配置 = 既不失灵活又驯服配置爆炸。

**一个 AFP 独有的理论红利**：npm 的粒度是为"人类装配工"优化的（人怕找和接线的麻烦，收敛到省人力的粒度）；而 AFP 里找和接线的成本主要由 **LLM** 承担，LLM 不怕接 200 个小块。**因此 AFP 的最优粒度可能系统性地比 npm 更细。** 这是一道前人问不出的开放研究题。

**补充：演化 vs 锁定的对抗。** 演化在早期自由、后期会被"锁定惯性"冻结——lodash 粒度不再变不是因为完美，而是 breaking change 成本太高没人敢动；一个被 100 个装配流依赖的装配块，即使该切细也没人敢动。对策：

> - **v1.0 前鼓励激进重构；v1.0 后走语义化版本，引擎必须支持「别名 / 多版本共存」**，让新旧粒度并行——收敛不是终点，**并行演化才是常态**。
> - **AFP 独有优势——版本迁移可自动化**：装配块是纯机制 + 强契约，schema diff 是机械的，引擎可 codemod 把配置从 `block@v1` 迁到 `v2`，而 LLM 恰好擅长这种机械迁移。这把锁定惯性进一步削弱。

### 由理论推出的愿景修正（重要）

把"机制可复用、策略不可复用"推到底，对 GitHub 愿景产生重大修正：

- **能在 GitHub 上全球复用的，只有"装配块（纯机制）"**——而这层其实早已存在，它就是 npm/库生态。
- **转接件、配置、数据是局部资产，天然不全球复用**（这正是业务软件复用 30 年失败的根因——业务各不相同）。

→ 所以"让 GitHub 提供无数可复用**业务流**"大概率是海市蜃楼，会撞上当年同一堵墙。**能共享的不是"开箱即用的业务流"，而是"装配块 + 转接模式 + 配置模板 + 起手式"（可复用的起点，不是可复用的成品）。**

> **结论性修正**：AssemFlow 生态的定位从"业务流市场"修正为 **「装配块 + 转接模式 + 配置模板的市场 + LLM 装配纪律」**。前者站不住，后者站得住。

### 理论血脉：面向特性编程（FOP）/ 软件产品线（SPL）

面向特性编程不是新名词，而是 1990 年代起的成熟谱系（Prehofer 1997 提出 FOP；上接 1990 年 SEI 的 FODA 特性建模，下接 FOSD / 软件产品线 SPL）。它对 AFP 既是富矿、也是警告。

**可直接借用（富矿）**：

- **特性模型（feature model）= AFP 配置层现成的严格形式化**：一棵带"必选/可选/或/互斥 + 跨树约束（requires/excludes）"的树，精确描述"哪些组合合法"。学术界早有**基于 SAT/CSP 的自动分析**（死特性检测、合法组合计数、改一个开关的影响面）——这正是「配置是可被机器导航的债」一节要的"从配置算影响面"，FOSD 里已解决，AFP 直接借、不必重造。
- **「问题空间 / 解空间 / 映射」三分法对上五元构件**：问题空间（业务要的特性）→ 解空间（装配块=纯机制）→ 映射（转接件 + 配置）。给了 AFP 一个被验证过的概念骨架。

**必须继承的两个警告**：

- **特性交互问题（feature interaction）**：两个单独都对的特性，组合起来会意外打架——这是 FOP 30 年未解的硬核，本质即「配置债」一节所述路径爆炸的深层形态。AFP 必撞，别幻想绕过；「静态可枚举」红线正是为压制它而设。
- **SPL 的采用杀手**：软件产线没普及，因前期"领域工程/特性建模"成本太高（与 MDA 同死法）。**而 AFP 真正可能的贡献，恰是用 LLM 把这道领域建模成本压下来。**

→ 一句定位：**AFP ≈ 特性模型（SPL）+ LLM 驱动的特性抽取 + 确定性装配。** FOP 给形式化与工具，LLM 补它当年没普及的成本。

## 验证方法

把"这条路走不走得通"拆成四个可证伪的维度，逐个查证而非臆断：

- 先例扫描：有没有人做过类似的？成败如何？
- 网络效应：GitHub"装配块飞轮"能否转起来？需要什么前置条件？
- 可靠性边界：确定性在哪类业务成立、在哪崩？
- AI 真实增量：LLM 是让老路第一次走通的关键变量，还是换皮？

## 维度一 · 先例扫描

**判断：这是个"一半很旧 + 一半很新"的组合。旧的一半失败过几十年；新的一半才是这次可能走通的关键。而且已有直接竞品。**

### 旧的一半：被追了 30 年的"积木复用梦"反复失败

- 可复用组件天生抽象，导致质量难工程化、难管理（[Why Software Reuse has Failed](https://www.dre.vanderbilt.edu/~schmidt/reuse-lessons.html)）。
- 复用的头号技术障碍是**"找不到、取不到合适的组件"**——不是造不出积木，是没人能在海量积木里找到对的那块（[Technical Impediments to Software Reuse](https://www.researchgate.net/publication/50366251_TECHNICAL_IMPEDIMENTS_TO_SOFTWARE_REUSE)）。知识制品的"过度充裕"本身反而让复用变成够不着的银弹（[CACM](https://cacm.acm.org/research/four-dynamics-for-bringing-use-back-into-software-reuse/)）。
- "组件市场能成为银弹"在 2003 年就被讨论过，结论是该产业当时太不成熟（[CACM 2003](https://cacm.acm.org/magazines/2003/8/6756-software-reuse-strategies-and-component-markets)）。

→ "让 GitHub 提供无数可靠装配块 + 可复用配置"正是这个**黑盒复用 + 开放组件市场**的老梦。它失败的不是技术，是"找得到、信得过、粒度对、有人维护"的生态难题。

### 低代码前车之鉴："10% 陷阱"

Thoughtworks 把低代码的根本局限称为**"10% 陷阱"**：90% 常见场景很省事，最后 10% 的定制/长尾要么做不出、要么比写代码还难（[Thoughtworks](https://www.thoughtworks.com/insights/podcasts/technology-podcasts/low-code-no-code-platforms-ten_percent-trap-limits-abstractions)）。这直接威胁"配置覆盖业务"和"非程序员产出代码"两个目标。

### 新的一半：LLM + 技能生态（2024 年后才出现的变量）

- **已有直接竞品**：Bit Cloud 的 Hope AI 被描述为"架构感知 agent，从 prompt、设计稿和已有可复用组件组装专业软件"（[chatgate.ai](https://chatgate.ai/post/bit-cloud)）——与本设想高度重合。
- "乐高原则：模块化可复用 AI 积木灵活组合"已被当作趋势（[xpert.digital](https://xpert.digital/en/reusable-ai-building-blocks/)）；"组件是开发者、非技术团队、AI 的共享语言"（[conf42](https://conf42.com/Machine_Learning_2025_Laly_BarIlan_language_components_understand)）。
- **最关键的新基建**：Anthropic 的 Claude Skills——"文件夹 + Markdown（YAML frontmatter + 指令）+ 可选脚本，按需加载"（[skywork](https://skywork.ai/blog/ai-agent/claude-skills-vs-mcp-vs-llm-tools-comparison-2025/)）——与本仓库 `claw/skills/` 结构一致；MCP 已成开放标准并捐给 Linux 基金会（[intuitionlabs](https://intuitionlabs.ai/articles/claude-skills-vs-mcp)）。

### 小结

装配流编程作为整体**不是蓝海**，竞品与趋势都在。问题不再是"有没有人想到"，而是"**LLM 时代为什么这次不会重蹈失败，你相对 Bit / Claude Skills / MCP 的楔子在哪**"。

## 维度二 · GitHub 网络效应能否成立

核心问题：软件**组件市场**失败了，但 npm / Docker Hub / Terraform Registry / Hugging Face 这些**开放注册表**成功了。差别在哪？AssemFlow 的"装配块飞轮"要成，必须满足成功者的共性条件。

### 成功注册表的共性（飞轮转起来的前提）

- **有一个主导工具/运行时把标准立起来**：npm 之所以爆发，是因为 Node.js 先创造了"在浏览器外跑 JS"的运行时，包管理器是顺势而生的配套（[TheServerSide](https://www.theserverside.com/blog/Coffee-Talk-Java-News-Stories-and-Opinions/The-secret-history-behind-the-success-of-npm-and-Node)）。**注册表不能脱离一个让它有用的运行时单独存在。**
- **复用单位有清晰契约 + 标准化打包**：能 `install` 一下就用，接口稳定。
- **跨平台一致性**：同一份制品到处能用，这点对面向全球的工具尤其关键（[Medium](https://medium.com/@miaoli1315/the-great-npm-shift-how-ai-tools-redefined-javascripts-package-manager-98963ea9d46b)）。
- **社区留存靠"社交 + 技术双重参与"**：会流失的贡献者特征是——不参与讨论、活跃度低、不长期投入（[ar5iv](https://ar5iv.labs.arxiv.org/html/1708.02618)）。光有仓库不够，得有活的社区。

### 对 AssemFlow 的推论

要让飞轮转起来，至少需要：

1. **先有"杀手级运行时/引擎"再谈注册表**。AssemFlow 引擎必须自己就足够有用（哪怕只服务一个用户），让装配块"装上就能跑"。否则就是空有市场没有 Node.js——必死。
2. **装配块要有强制契约**：严格 I/O Schema + 版本 + 测试用例，否则退化成又一个"找不到、信不过"的代码堆。
3. **GitHub 只是托管层，不是飞轮本身**。GitHub 提供版本/PR/Actions，但"发现—信任—装配"这套机制得 AssemFlow 自己造（语义检索 + 校验 + 评分）。

### 反向风险

- **锁定与治理**：注册表天然有制造锁定的动机，哪怕非营利的也有（[Scarf](https://about.scarf.sh/post/package-registries-and-open-source/)）。开放生态的治理规则要早定。
- **冷启动悖论（必须拆成两个分别解）**：
  - **验证冷启动**（证明范式在单点根本成立）——**现在就解**，越小越干净越好。
  - **生态冷启动**（注册表飞轮自转）——**现在别碰，甚至可能永远不需要**：AFP 哪怕只作为"一个团队内部的装配纪律"也可能有价值。
  别让"生态冷启动"这个大难题，绑架"验证冷启动"这个小目标。
- **npm 的真正教训是"排序"而非"规模"**：npm 不是靠"挑小项目起步"，而是 Node.js 这个杀手级运行时先创造需求、注册表搭车长出来（见「成功注册表的共性」）。所以正确问题不是"大项目还是小项目"，而是"**没有任何生态时，AssemFlow 引擎对单一用户（你）有用的最小杀手场景是什么**"——单机有用先于多人有用。
- **clawerworld 的定位修正**：clawerworld 的核心（闯关/竞技场/河谷/agent）是高度定制、叙事化、强状态的"10% 区"，**不宜做"甜区证明"的试验田**（等于专挑最难处开局）；但它的真实烂摊子正好适合做**"边界证伪"的试验田**。结论：clawerworld 顶多是"真实甜区流程的素材来源 + 边界试验田"，**不是生态供给起点**。

## 维度三 · 可靠性边界

核心问题：AFP 主打"确定性/可靠"。但 LLM 恰恰是不确定的。边界在哪？

### LLM 直接写依赖/代码的真实风险（必须正视）

- 研究显示，约 **20%** 的 LLM 生成代码样本（57.6 万个 Python/JS 样本）推荐了**不存在的包**（[BleepingComputer](https://www.bleepingcomputer.com/news/security/ai-hallucinated-code-dependencies-become-new-supply-chain-risk/)，原始论文 [arXiv 2406.10279](https://arxiv.org/html/2406.10279)）。
- 由此催生新型供应链攻击 **"slopsquatting"**：攻击者抢注 AI 爱幻觉出来的包名投毒（[Trend Micro](https://www.trendmicro.com/vinfo/no/security/news/cybercrime-and-digital-threats/slopsquatting-when-ai-agents-hallucinate-malicious-packages)、[SecurityWeek](https://www.securityweek.com/ai-hallucinations-create-a-new-software-supply-chain-threat/)）。
- 2025-2026 的供应链攻击已升级为**跨 npm/PyPI/Docker Hub/Actions 同时下手**（[Aquilax](https://aquilax.ai/blog/supply-chain-worms-multi-ecosystem)）。

→ 结论：**"让 AI 自由写代码并直接拉依赖"本身就是高风险路径。** 这恰好反证了 AFP 的价值主张是对的方向——但前提是它真的把 AI 关在笼子里。

### 确定性成立 / 崩塌的分界

- **成立区**：业务流程结构稳定、可枚举（注册、退款、审核、CRUD、定时任务）。这类正是 AFP 的甜区，也是低代码 90% 能覆盖的部分。
- **崩塌区**：高度定制的算法、复杂 UI 交互、强耦合遗留系统改造、需求长尾——即低代码的"10% 陷阱"。AFP 在这里同样会碰壁。

### 设计上的硬约束（让"可靠"不沦为口号）

1. **AI 只在设计期产配置，运行期拼装零 AI**——这是确定性 NFR 能成立的唯一前提。
2. **依赖白名单 + 制品校验**：装配块的依赖必须白名单内、可审计，从源头掐死 slopsquatting。
3. **生成产物必须过编译 + 跑归档测试**才算交付，杜绝"看起来对"的幻觉代码。
4. **明确兜底**：落入"10% 区"时不硬装，回退到人工/对话式开发并引导沉淀为新装配块。

## 维度四 · AI 的真实增量

**这是全篇最关键的洞察。**

LLM 恰好破解了让"积木复用梦"失败的**两个历史死结**：

| 历史死结 | LLM 带来的解法 |
| :--- | :--- |
| ①"找不到合适的组件"（复用头号障碍） | LLM 语义匹配天然解决"发现"——自然语言 → 装配块/装配流 |
| ②"积木太死，最后 10% 做不出"（低代码陷阱） | LLM 能适配、补胶水、填空，绕过僵硬边界 |

**但矛盾在于**：让老梦复活的恰恰是 LLM，而 LLM 又是破坏"确定性"的元凶（见「可靠性边界」中的幻觉数据）。

**化解之道 = AFP 的真正楔子**：把 LLM 严格限定在"发现 + 配置 + 适配"这一侧（设计期、有人审），把"执行"留在确定性纯代码这一侧（运行期、零 AI）。

这条边界，正是 AFP 区别于竞品的地方：
- 别人（Bit Hope AI、vibe coding）是"**AI 直接组装并产出/运行代码**"——快，但继承了 LLM 的不确定与供应链风险。
- AFP 主张"**AI 只产出可审计的确定性配置，执行不碰 AI**"——慢一点，但可复现、可审计、可治理。

→ AI 不是换皮，它是让这条 30 年老路第一次有机会走通的关键变量。但能不能走通，取决于你**守不守得住那条"AI 只管配置"的边界**——一旦为了图快让 AI 直接产代码，AFP 就退化成又一个 vibe coding。

## 综合裁决

| 问题 | 结论 |
| :--- | :--- |
| 思想是否原创？ | **否**。方向有竞品（Bit）、有趋势（乐高式 AI 积木）、有基建（Claude Skills/MCP）。 |
| 思想是否成立？ | **有条件成立**。它押注的"LLM 解决发现+适配"是真实的、新出现的红利。 |
| 是否可靠？ | **取决于边界纪律**。守住"AI 只管配置、运行期零 AI + 白名单 + 编译/测试关"才可靠；否则退化为 vibe coding。 |
| 最大死因风险？ | **冷启动悖论**（没库没人用）+ **10% 陷阱**（长尾失效）+ **边界失守**（图快让 AI 直接写代码）。 |
| 差异化楔子？ | **确定性执行**：AI 产配置、引擎确定拼装、全程可审计。这是相对竞品唯一站得住的点。 |

一句话：**装配流编程不是"新发明"，而是一个"旧梦 + 新变量"的赌注。赌注能否赢，不在技术新不新，而在你能否守住"确定性"这条别人不愿守的纪律，并解决冷启动。**

## 建议的最小验证实验（证伪优先，不是实施）

在投入造引擎/生态之前，用最小代价先回答"它到底成不成立"：

1. **甜区证伪（用全新极小项目，不用 clawerworld）**：建一个独立微项目（如"注册 + 发邮件 + 写审计日志"服务），手写 3 个装配块 + 1 份配置，纯脚本拼装，验证"只改配置就能出可编译+过冒烟的代码"。用全新项目是为了零遗留、零叙事耦合，信号最纯。成立 → 甜区假设为真。
2. **10% / 边界证伪（用 clawerworld 的烂摊子）**：故意挑 clawerworld 一个偏定制的真实需求，看 AFP 在哪一步崩、崩得多难看。崩点位置决定 AFP 的适用边界。
3. **复用证伪**：让第二个业务流（如带短信验证的注册）复用第一个的装配块，**不改装配块代码、只加配置/转接件**。能复用 → 复用假设为真；要改代码 → 粒度假设证伪。

这三个实验任何一个失败，都比写一份漂亮 PRD 更有价值——它们直接告诉你这条路在哪段是实的、在哪段是空的。
